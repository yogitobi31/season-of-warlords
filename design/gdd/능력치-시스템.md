# 능력치 시스템 (Stat System)

> **Status**: In Design
> **Author**: Juwon + Claude Code agents
> **Last Updated**: 2026-04-17
> **Implements Pillar**: Pillar 2 (Visible Snowball), Pillar 4 (Small but True)

## Overview

능력치 시스템은 유랑단의 모든 캐릭터(플레이어, 동료, 적)의 수치 속성을 정의하는 데이터 레이어다. 체력(HP), 공격력(ATK), 방어력(DEF), 이동 속도(SPD) 등의 기본 스탯을 각 캐릭터 리소스에 저장하고, 이를 필요로 하는 시스템(체력/데미지, 동료 AI, 경험치/성장)에 단일 진실 출처로 제공한다. 플레이어는 이 시스템을 직접 느끼지 않는다 — 동료가 합류할 때 전투가 눈에 띄게 달라지는 것, 레벨이 오를수록 적이 더 쉽게 쓰러지는 것이 이 시스템의 존재를 증명한다.

## Player Fantasy

플레이어는 이 시스템을 보지 않는다. 느끼지도 않는다 — 제대로 작동할 때는.

이 시스템이 실패하면 플레이어가 느끼는 것: 어제 합류한 동료가 아무 차이도 만들지 못한다 (Pillar 2 실패). 모든 동료가 같은 무게로 느껴진다 (Pillar 4 실패). 능력치 시스템은 유랑단이 커가는 느낌의 토대이자, 사람마다 다른 무게가 숫자로 새겨지는 곳이다.

## Detailed Design

### Core Rules

1. **모든 캐릭터는 동일한 `CharacterStats` Resource를 사용한다.** 플레이어, 동료, 적 간의 스탯 구조 차이 없음. 차이는 값으로 표현한다.

2. **MVP 스탯은 4개다:** `max_hp` (int), `atk` (int), `def` (int), `spd` (float, px/s). 이 4개 이외의 스탯은 해당 스탯을 필요로 하는 시스템 GDD가 작성될 때까지 추가하지 않는다.

3. **두 층 구조 (base + modifier):**
   - `base_*` — 영구적 기저값. 레벨업과 동료 성장 시스템만 여기에 쓴다.
   - `mod_*` — 임시 수정값. 버프/디버프 등 전투 효과가 여기에 쓴다. MVP에서는 항상 0.
   - `computed` 프로퍼티 (`max_hp`, `atk`, `def`, `spd`) — `base_* + mod_*`를 반환. 다운스트림 시스템은 이것만 읽는다.

4. **스탯 변경 시 signal을 발생시킨다.** `stats_changed(stat_name: String)`. UI와 AI 시스템은 매 프레임 polling하지 않고 이 signal에 반응한다.

5. **캐릭터 템플릿은 `.tres` 파일로 저장한다.** 런타임에서는 `stats_template.duplicate()` 로 인스턴스를 생성해 원본을 보호한다.

6. **`GrowthRate`는 별도 Resource다.** CharacterStats와 분리하여 경험치/레벨업 시스템만 읽는다.

### States and Transitions

능력치 시스템은 상태 머신이 아닌 데이터 레이어다. 다만 인스턴스 생명주기가 있다:

| 단계 | 설명 | 소유자 |
|------|------|--------|
| **Template** | `.tres` 파일로 저장된 읽기 전용 기저 스탯 | Assets |
| **Runtime Instance** | `duplicate()`로 생성된 런타임 복사본 | 각 캐릭터 노드 |
| **Modified** | `mod_*` 값이 변경된 상태 (버프 등 적용) | 전투 시스템 |
| **Leveled** | `base_*` 값이 증가한 상태 (레벨업 후) | 경험치/레벨업 시스템 |

### Interactions with Other Systems

| 방향 | 시스템 | 읽는/쓰는 값 | 인터페이스 |
|------|--------|-------------|-----------|
| **→ (출력)** | 체력/데미지 시스템 | `max_hp`, `atk`, `def` | computed 프로퍼티 직접 읽기 |
| **→ (출력)** | 동료 AI 시스템 | `atk`, `spd` | computed 프로퍼티 직접 읽기 |
| **→ (출력)** | 플레이어 캐릭터 컨트롤러 | `max_hp`, `spd` | computed 프로퍼티 직접 읽기 |
| **← (입력)** | 경험치/레벨업 시스템 | `base_max_hp`, `base_atk`, `base_def`, `base_spd` 증가 | `base_*` 직접 쓰기, `stats_changed` emit |
| **← (입력)** | 전투 시스템 (VS+) | `mod_atk`, `mod_def` 임시 수정 | `mod_*` 직접 쓰기, `stats_changed` emit |

## Formulas

### Formula 1 — Computed Stat (공통 패턴)

`effective_stat = base_stat + mod_stat`

| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| 기저값 | `base_stat` | int / float | 스탯별 최솟값 이상 | 영구값. 레벨업 시스템만 쓴다 |
| 수정값 | `mod_stat` | int / float | 음수 가능 | 임시값. 버프/디버프. MVP에서 항상 0 |
| 실제값 | `effective_stat` | int / float | 스탯별 하한으로 클램프 | 다운스트림 시스템이 읽는 값 |

**클램핑 조건:**

| 스탯 | 최솟값 | 최댓값 | 이유 |
|------|--------|--------|------|
| `max_hp` | 1 | 9999 | 0이면 선택 로직 예외; 9999 초과는 UI 오버플로 |
| `atk` | 0 | 999 | atk=0 허용 (무기 해제 여지) |
| `def` | 0 | 999 | def ≥ 적 atk이면 피해 0 → 무한 전투 루프 위험. 데미지 시스템 GDD에서 최소 데미지 보장 조건으로 재검토 필요 |
| `spd` | 10.0 | 600.0 | 10 미만: AI pathfinding 타임아웃 가능; 600 초과: 소형 맵에서 프레임당 충돌 감지 스킵 가능 |

**예시:**
```
base_def = 5, mod_def = -8  → effective_def = max(0, 5 + (-8)) = 0
base_spd = 130.0, mod_spd = 0.0  → effective_spd = 130.0
```

---

### Formula 2 — 초기값 (MVP 제안값)

> ⚠️ 아래 수치는 `damage = max(1, atk - def)` 단순 감산 공식을 가정한 참조값입니다. 체력/데미지 시스템 GDD에서 공식이 확정되면 재검토가 필요합니다.

**플레이어 캐릭터:**

| 스탯 | 제안값 |
|------|--------|
| `base_max_hp` | 80 |
| `base_atk` | 20 |
| `base_def` | 8 |
| `base_spd` | 130.0 px/s |

**동료 3인 프로파일:**

| 유형 | `base_max_hp` | `base_atk` | `base_def` | `base_spd` |
|------|---------------|------------|------------|------------|
| 탱커형 | 140 | 14 | 18 | 90.0 |
| 균형형 | 90 | 18 | 10 | 120.0 |
| 속도형 | 60 | 22 | 4 | 160.0 |

**기본 적 범위:**

| 강도 | `max_hp` | `atk` | `def` | `spd` |
|------|----------|-------|-------|-------|
| 약한 적 | 30~40 | 8~10 | 0~3 | 70~90 |
| 보통 적 | 55~70 | 12~16 | 4~8 | 90~110 |
| 강한 적 | 120~180 | 20~28 | 10~15 | 50~80 |

**Output Range:** 스탯별 클램프 조건 이내. 팀 합류 전후 체감: 솔로 총 HP(80) → 탱커 합류(220, +175%) — Pillar 2 시각적 성장감 목표.

---

### Formula 3 — GrowthRate (레벨업 1회당 증가량, VS 단계 사용)

`new_base_stat = base_stat + growth_value_per_level`

| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| 현재 기저값 | `base_stat` | int / float | 스탯별 최솟값 이상 | 레벨업 직전 현재 기저값 |
| 레벨당 증가량 | `growth_value_per_level` | int / float | 아래 표 참조 | GrowthRate 리소스에 저장 |

**동료 유형별 성장률 (1레벨당):**

| 유형 | `hp_growth` | `atk_growth` | `def_growth` | `spd_growth` |
|------|-------------|--------------|--------------|--------------|
| 탱커형 | 14~18 | 1~2 | 3~4 | 0.0~1.0 |
| 균형형 | 8~10 | 2~3 | 1~2 | 1.0~2.0 |
| 속도형 | 4~6 | 3~5 | 0~1 | 2.0~4.0 |

**예시 (속도형, 레벨 1→5, hp_growth=5, atk_growth=4, def_growth=1, spd_growth=3.0):**
```
base_max_hp: 60 → 60 + (5×4) = 80
base_atk:    22 → 22 + (4×4) = 38
base_def:     4 →  4 + (1×4) = 8
base_spd:  160.0 → 160.0 + (3.0×4) = 172.0
```

## Edge Cases

**스탯 경계 케이스:**

- **If `effective_def = 0` (base_def=0, mod_def=0)**: 적의 ATK가 그대로 피해로 적용된다. 의도된 동작 — "방어가 없는 캐릭터는 모든 피해를 받는다." 데미지 시스템 GDD는 이 상태를 정상 입력으로 처리해야 한다.

- **If `effective_atk = 0`**: 데미지 공식이 이 경우에도 최소 1 피해를 보장해야 한다. 보장 여부는 본 시스템이 아닌 체력/데미지 시스템 GDD의 책임이다.

- **If `base_spd + mod_spd < 10.0`** (디버프로 SPD가 하한 미만으로 내려가려는 경우): computed getter에서 `clamp(result, 10.0, 600.0)`을 적용하여 10.0으로 고정한다. `base_spd` 자체는 변경하지 않는다.

- **If `effective_def ≥ enemy_atk`**: 데미지가 0 이하가 되어 무한 전투 루프 위험. 체력/데미지 시스템 GDD는 최소 1 피해를 보장하는 조건을 명시해야 한다 — 이 의존성을 Dependencies 섹션에 기재한다.

**동시성/순서 케이스:**

- **If 버프와 디버프가 같은 프레임에 `mod_atk`에 쓰이는 경우**: GDScript 싱글스레드이므로 순차 실행된다. `stats_changed` signal이 두 번 발생해 UI가 중간 상태를 1프레임 노출할 수 있다. MVP에서 mod_*는 항상 0이므로 실질 문제 없음. VS+ 단계에서 버프 시스템 GDD가 실행 순서를 명시해야 한다.

- **If 전투 중 레벨업이 발생하여 `base_atk`가 변경되는 경우**: 현재 진행 중인 공격 계산은 변경 전 값을 사용한다. 다음 공격 계산부터 새 값이 적용된다. 의도된 동작.

- **If `stats_changed` signal 핸들러 안에서 `base_*` 또는 `mod_*`를 변경하는 코드가 연결되는 경우**: signal 재진입으로 `stats_changed`가 연속 발생할 수 있다. signal 핸들러는 `base_*`나 `mod_*`를 쓰지 않는다는 규칙을 모든 다운스트림 시스템이 지켜야 한다.

**`duplicate()` 생명주기 케이스:**

- **If 같은 `.tres` 템플릿에서 두 동료를 `duplicate()`로 생성하는 경우**: `CharacterStats`의 모든 프로퍼티가 int/float 프리미티브이므로 shallow copy로도 두 인스턴스의 독립성이 보장된다. 단, 미래에 `CharacterStats` 내부에 배열이나 Resource 타입을 추가할 경우 `duplicate(true)` (deep copy)로 변경해야 한다.

- **If 동료가 파티에서 이탈하거나 사망하는 경우**: 파티 매니저는 해당 동료의 `CharacterStats` 참조를 즉시 배열에서 제거해야 한다. 제거하지 않으면 stale reference가 남아 해제된 노드의 인스턴스가 메모리에 유지된다.

**GrowthRate 케이스:**

- **If `growth_value_per_level = 0`인 스탯에서 레벨업이 발생하는 경우** (예: 탱커형 spd_growth=0): 기저값에 변화 없음. 레벨업 시스템은 이 경우 해당 스탯의 `stats_changed`를 emit하지 않거나, 연출 레이어에서 delta=0이면 "변화 없음"으로 표시한다. 어떻게 처리할지는 레벨업 연출 GDD에서 결정한다.

- **If 레벨업으로 `base_max_hp`가 9999를 초과하려는 경우**: 레벨업 시스템이 `base_max_hp = min(base_max_hp + hp_growth, 9999)`로 직접 클램프하여 `base_*`와 `effective_*`의 불일치를 원천 차단한다. `computed` getter의 클램프만 의존하지 않는다.

**MVP 특이 케이스:**

- **If MVP에서 코딩 실수로 `mod_*`에 값이 쓰이는 경우**: `stats_changed` signal이 발생하고 버프가 걸리지 않았는데 스탯이 올라간 것처럼 동작한다. 발견하기 어려운 silent fail이다. MVP 빌드에서는 `mod_*` setter에 `push_warning()`을 추가하여 의도치 않은 쓰기를 경고한다.

- **If 동료 합류 시점에 `CharacterStats` 인스턴스를 언제 생성하는가**: 영입 완료(합류 확정) 시점에 `duplicate()`를 호출한다. 영입 퀘스트 진행 중에는 `.tres` 템플릿을 직접 참조하여 "예상 스탯"을 표시한다. 이 시점 결정은 동료 영입 퀘스트 GDD에서 확정한다.

## Dependencies

**이 시스템이 의존하는 것 (Upstream):**

없음. 능력치 시스템은 Foundation 레이어다 — 다른 시스템에서 데이터를 받지 않는다.

**이 시스템에 의존하는 것 (Downstream):**

| 시스템 | 의존 유형 | 읽는 값 | 의존 강도 |
|--------|-----------|---------|---------|
| 체력/데미지 시스템 | Hard | `max_hp`, `atk`, `def` | 없으면 데미지 계산 불가 |
| 동료 AI 시스템 | Hard | `atk`, `spd` | 없으면 AI 행동 결정 불가 |
| 플레이어 캐릭터 컨트롤러 | Hard | `max_hp`, `spd` | 없으면 이동 속도, HP 상한 없음 |
| 경험치/레벨업 시스템 | Hard | `base_*` (쓰기) | 없으면 레벨업 결과 저장 불가 |
| 동료 성장 시스템 | Hard | `base_*` (쓰기), GrowthRate 읽기 | 없으면 동료 성장 불가 |

**전이적 의존성 (검증 필요):**

체력/데미지 시스템 GDD가 완성되면 — 이 시스템의 `def` 클램프(min=0)와 "최소 1 피해 보장" 조건이 일치하는지 확인. 불일치 시 Edge Cases에 기술된 무한 전투 루프 위험이 현실화된다.

## Tuning Knobs

| 튜닝 놉 | 현재 값 | 안전 범위 | 너무 높으면 | 너무 낮으면 | 상호작용 |
|---------|---------|---------|-----------|-----------|---------|
| 플레이어 `base_max_hp` | 80 | 50~150 | 긴장감 감소 (적 단타로 안 죽음) | 2~3타에 사망 | 약한 적 atk와 함께 조정 |
| 플레이어 `base_atk` | 20 | 10~40 | 약한 적 1타 처치 → 보상감 감소 | 5타 이상 필요 → 전투 지루함 | 약한 적 max_hp와 함께 조정 |
| 플레이어 `base_spd` | 130.0 | 80~200 | 맵 횡단 너무 빠름, 탐험 긴장감 감소 | 이동이 답답하게 느껴짐 | 맵 크기와 함께 조정 |
| 탱커 `base_max_hp` | 140 | 100~200 | 무적 체감, 플레이어가 탱커 뒤에만 숨음 | 플레이어와 차이 안 느껴짐, 탱커 정체성 약화 | 플레이어 max_hp 대비 1.5~2.5배 유지 |
| `spd` 하한 | 10.0 | 5.0~30.0 고정 | 캐릭터 사실상 정지 가능 | AI pathfinding 타임아웃 가능 | 내리지 않는 것을 권장 |
| `max_hp` 상한 | 9999 | 고정 | UI 오버플로 | — | UI 자릿수 설계와 연동 |
| GrowthRate `hp_growth` (탱커) | 14~18 | 8~25 | 레벨업마다 HP 급증 → 밸런스 붕괴 | 성장 체감 없음 | 레벨 캡 확정 후 재조정 |

**상호작용 주의:**
- `base_atk`와 `base_def`는 단독 조정하면 안 된다. 약한 적의 `max_hp`와 함께 "약한 적을 3~5타에 처치"라는 전투 속도 목표를 기준으로 조정한다.
- 동료 `base_spd`는 플레이어 `base_spd`보다 낮아야 한다 (플레이어 조작감 우선). 현재 플레이어 130 > 균형형 120 > 탱커 90이 그 원칙을 따른다.

### Core Rule 조건

**AC-STAT-CR-01** (공용 스탯 구조체)
- **GIVEN** 플레이어, 동료, 적 캐릭터 노드가 씬에 인스턴스화되어 있고
- **WHEN** 각 노드에서 CharacterStats Resource 타입을 확인하면
- **THEN** 세 노드 모두 동일한 `CharacterStats` 클래스 타입을 반환해야 한다. 플레이어 전용 또는 적 전용 스탯 클래스가 별도로 존재하면 실패.

**AC-STAT-CR-02** (MVP 스탯 4개)
- **GIVEN** `CharacterStats` 인스턴스가 생성되어 있고
- **WHEN** 해당 인스턴스의 exported 프로퍼티 목록을 검사하면
- **THEN** `base_max_hp` (int), `base_atk` (int), `base_def` (int), `base_spd` (float), `mod_max_hp` (int), `mod_atk` (int), `mod_def` (int), `mod_spd` (float)가 존재해야 한다. 그 외 스탯 프로퍼티(예: `crit`, `res`)가 존재하면 실패.

**AC-STAT-CR-03** (computed 프로퍼티가 base + mod를 반환)
- **GIVEN** `CharacterStats` 인스턴스에서 `base_atk = 20`, `mod_atk = 5`로 설정했을 때
- **WHEN** `atk` computed 프로퍼티를 읽으면
- **THEN** 반환값이 정확히 `25`여야 한다.

**AC-STAT-CR-04** (stats_changed signal 발생)
- **GIVEN** `CharacterStats` 인스턴스에 `stats_changed` signal 수신자가 연결되어 있고
- **WHEN** `base_atk` 값을 `20`에서 `25`로 변경하면
- **THEN** `stats_changed` signal이 정확히 1회 발생하고, `stat_name` 인자 값이 `"atk"`여야 한다.

**AC-STAT-CR-05** (duplicate()로 독립 인스턴스 생성)
- **GIVEN** 동일한 `.tres` 템플릿에서 `duplicate()`를 두 번 호출하여 인스턴스 A와 B를 생성하고
- **WHEN** 인스턴스 A의 `base_atk`를 `99`로 변경하면
- **THEN** 인스턴스 B의 `base_atk`는 원본 템플릿 값을 유지해야 한다.

### Formula 조건

**AC-STAT-FM-01** (양수 modifier 합산)
- **GIVEN** `base_def = 5`, `mod_def = 3`일 때
- **WHEN** `def` computed 프로퍼티를 읽으면
- **THEN** 반환값이 `8`이어야 한다.

**AC-STAT-FM-02** (음수 modifier — def 클램프 하한)
- **GIVEN** `base_def = 5`, `mod_def = -8`일 때
- **WHEN** `def` computed 프로퍼티를 읽으면
- **THEN** 반환값이 `0`이어야 한다. `-3`이 반환되면 실패.

**AC-STAT-FM-03** (spd 클램프 하한)
- **GIVEN** `base_spd = 20.0`, `mod_spd = -15.0`일 때
- **WHEN** `spd` computed 프로퍼티를 읽으면
- **THEN** 반환값이 `10.0`이어야 하고, `base_spd`는 `20.0`을 유지해야 한다.

**AC-STAT-FM-04** (max_hp 클램프 상한)
- **GIVEN** `base_max_hp = 9990`, `mod_max_hp = 20`일 때
- **WHEN** `max_hp` computed 프로퍼티를 읽으면
- **THEN** 반환값이 `9999`여야 한다.

**AC-STAT-FM-05** (max_hp 클램프 하한)
- **GIVEN** `base_max_hp = 1`, `mod_max_hp = -5`일 때
- **WHEN** `max_hp` computed 프로퍼티를 읽으면
- **THEN** 반환값이 `1`이어야 한다.

**AC-STAT-FM-06** (플레이어 초기값)
- **GIVEN** 플레이어 `.tres` 템플릿 (TBD — 충돌 연출 시스템 GDD 작성 시 경로 확정)을 `duplicate()`로 인스턴스화하고 `mod_*` 모두 `0`인 상태에서
- **WHEN** `max_hp`, `atk`, `def`, `spd` computed 프로퍼티를 읽으면
- **THEN** `max_hp = 80`, `atk = 20`, `def = 8`, `spd = 130.0`이어야 한다.

### Edge Case 조건

**AC-STAT-EC-01** (spd 디버프 시 base_spd 불변)
- **GIVEN** `base_spd = 15.0`이고 `mod_spd = -20.0`이 할당된 상태에서
- **WHEN** `spd` computed 프로퍼티를 읽으면
- **THEN** 반환값이 `10.0`이어야 하고, `base_spd`는 `15.0`을 유지해야 한다.

**AC-STAT-EC-02** (레벨업으로 base_max_hp 9999 초과 시도)
- **GIVEN** `base_max_hp = 9995`이고 레벨업 시스템이 `hp_growth = 10`을 더하려는 상황에서
- **WHEN** 레벨업 시스템의 stat 업데이트 로직이 실행되면
- **THEN** `base_max_hp`의 저장값이 `9999`여야 한다. `10005`로 저장되어 있으면 실패 (computed getter 클램프만으로 통과 불가 — `base_*` 자체가 9999 이하여야 함).

**AC-STAT-EC-03** (duplicate() 인스턴스 독립성 — mod 값 포함)
- **GIVEN** 동일한 템플릿에서 인스턴스 A와 B를 `duplicate()`로 생성하고
- **WHEN** 인스턴스 A의 `mod_atk`를 `50`으로 변경하면
- **THEN** 인스턴스 B의 `mod_atk`는 `0`을 유지하고, `atk` computed 프로퍼티는 원본 `base_atk` 값을 반환해야 한다.

---

### 크로스-시스템 인터랙션 조건

**AC-STAT-XS-01** (stats_changed signal이 수신자에게 전달됨)
- **GIVEN** `stats_changed` signal에 테스트용 수신 함수가 연결되고, 수신 횟수 카운터가 `0`으로 초기화되어 있으며
- **WHEN** `base_max_hp`를 `80`→`100`으로, 이어서 `base_atk`를 `20`→`25`로 변경하면
- **THEN** 카운터가 정확히 `2`가 되어야 하고, 수신된 `stat_name`이 차례로 `"max_hp"`, `"atk"`여야 한다.
