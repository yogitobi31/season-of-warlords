# 퀘스트 상태 머신 (Quest State Machine)

> **Status**: Done
> **Author**: Juwon + agents
> **Last Updated**: 2026-04-26
> **Implements Pillar**: Pillar 1 (모인 사람들) — 동료를 얻는 과정의 구조적 뼈대

## Overview

퀘스트 상태 머신은 각 동료 영입 퀘스트의 진행 상태를 추적하고 완료 조건을 판정하는 Feature 레이어 시스템이다. 대화 시스템에서 이벤트를 수신해 퀘스트를 시작하고, 인벤토리/전투 결과를 모니터링해 조건 충족 여부를 판단하며, 완료 시 NPC 상태를 갱신한다.

MVP에서 각 퀘스트는 3가지 상태만 가진다:
```
NOT_STARTED ──대화 이벤트──► ACTIVE ──조건 충족──► COMPLETED
```

**퀘스트 정의 구조:**
```gdscript
quest = {
    "id": "quest_blacksmith",
    "conditions": [
        { "type": "has_item", "item_id": "iron_ore", "amount": 3 }
    ],
    "on_complete": {
        "npc_id": "blacksmith",
        "npc_state": RelationshipState.COMPANION,   # NPC-상태-관리.md 권위 enum
        "remove_items": [                            # 아이템-데이터베이스.md 계약 1 이행
            { "item_id": "iron_ore", "amount": 3 }
        ]
    }
}
```

## Player Fantasy

퀘스트 목표가 명확하고 진행이 느껴져야 한다. "철광석 3개를 모아야 한다" → 하나씩 모을 때마다 가까워지는 느낌. 마지막 조건이 충족되는 순간 "드디어!"라는 감각. 이 시스템은 그 순간을 만드는 판정 엔진이다.

## Detailed Rules

**R-1: QuestManager Autoload**
`QuestManager`로 등록. 모든 퀘스트 상태를 `quests: Dictionary` (`quest_id → QuestState`)로 관리. 씬 전환 시에도 유지.

**R-2: 퀘스트 시작**
`start_quest(quest_id: String)` — 대화 시스템의 `on_complete` 이벤트로 호출. `NOT_STARTED → ACTIVE`. 이미 `ACTIVE` 또는 `COMPLETED`면 무시.

**R-3: 조건 타입 (MVP)**

| 타입 | 판정 방법 | 예시 |
|------|-----------|------|
| `has_item` | `Inventory.has_item(item_id, amount)` | 철광석 3개 보유 |
| `enemy_killed` | 적 사망 신호 카운팅 | 슬라임 5마리 처치 |
| `location_visited` | 씬 전환 완료 신호 수신 | 광산 방문 |

**R-4: 조건 확인 시점**
조건 타입별로 확인 시점이 다름:
- `has_item` → 인벤토리 `item_added` 신호 수신 시마다 재확인
- `enemy_killed` → 적 `health_depleted` 신호 수신 시마다 카운트
- `location_visited` → 씬 전환 `transition_completed` 신호 수신 시 확인

**R-5: 퀘스트 완료**
모든 조건 충족 시 → `ACTIVE → COMPLETED`. `on_complete` 순서대로 실행:
1. `NPCRegistry.set_state(npc_id, RelationshipState.COMPANION)` — NPC 상태 갱신
2. `on_complete.remove_items` 배열 순회 → `Inventory.remove_item(item_id, amount)` — 퀘스트 아이템 제거 (아이템-데이터베이스.md 계약 1)
3. `quest_completed(quest_id)` 신호 발행

**R-6: 다중 조건**
모든 조건이 동시에 충족되어야 완료. 조건 배열은 AND 관계.

## Formulas

수학 공식 없음. `enemy_killed` 카운터만 정의.
```
kill_count[quest_id][enemy_type] += 1
condition_met = kill_count[quest_id][enemy_type] >= required_count
```

## Edge Cases

**EC-1: 퀘스트 시작 전 조건 아이템 보유**
`start_quest()` 호출 시 즉시 조건 확인. 이미 아이템이 있으면 즉시 완료 가능.

**EC-2: COMPLETED 퀘스트 재시작 시도**
R-2에 정의됨 — 무시.

**EC-3: 퀘스트 미시작 상태에서 조건 아이템 수집**
카운팅 안 함. `ACTIVE` 상태에서만 조건 모니터링.

**EC-4: 존재하지 않는 `quest_id`로 `start_quest()` 호출**
경고 로그 + 무시.

**EC-5: NPC 상태 갱신 실패 (`npc_id` 없음)**
경고 로그. 퀘스트는 `COMPLETED`로 마킹되지만 동료 합류 불발. 설계 오류로 간주.

## Dependencies

**이 시스템이 의존하는 것 (Upstream)**
- **NPC 상태 관리** — `NPCRegistry.set_state()` 호출, `on_complete` 실행
- **인벤토리 시스템** — `has_item()`, `item_added` 신호 구독
- **대화 시스템** — `on_complete` 이벤트로 `start_quest()` 트리거
- **씬 전환 시스템** — `transition_completed` 신호로 `location_visited` 판정

**이 시스템에 의존하는 것 (Downstream)**

| 시스템 | 의존 내용 |
|--------|-----------|
| **동료 영입 퀘스트** | `quest_completed` 신호 수신 → 동료 합류 이벤트 트리거 |
| **세이브/로드 시스템** | `quests` 딕셔너리 저장/복원 |

## Tuning Knobs

조건 수치는 퀘스트 데이터 파일에서 관리. 이 시스템은 판정 엔진이고 수치는 직접 정의하지 않음. 각 퀘스트의 요구 아이템 수량/처치 수는 `data/quests/` 리소스 파일에서 조정.

## Acceptance Criteria

| # | 조건 | 검증 방법 |
|---|------|-----------|
| AC-1 | `start_quest(id)` 호출 시 상태가 `ACTIVE`로 변경된다 | 단위 테스트 |
| AC-2 | `has_item` 조건 충족 시 퀘스트가 `COMPLETED`로 변경된다 | 단위 테스트 |
| AC-3 | `enemy_killed` 조건: 요구 수 미만이면 `ACTIVE` 유지, 충족 시 `COMPLETED` | 단위 테스트 |
| AC-4 | 완료 시 지정된 NPC 상태가 `RelationshipState.COMPANION`으로 갱신된다 | 단위 테스트 |
| AC-5 | `COMPLETED` 퀘스트에 `start_quest()` 재호출 시 상태 변화 없음 | 단위 테스트 |
| AC-6 | `quest_completed(quest_id)` 신호가 완료 시 발행된다 | 단위 테스트 |
| AC-7 | 완료 시 `on_complete.remove_items`에 정의된 아이템이 인벤토리에서 제거된다 | 단위 테스트 |
