# 동료 합류 이벤트 (Companion Join Event)

> **Status**: In Design
> **Author**: Juwon + agents
> **Last Updated**: 2026-04-26
> **Implements Pillar**: Pillar 1 (모인 사람들) + Pillar 2 (눈에 보이는 성장) — 영입의 결실이 화면에 나타나는 순간

## Overview

동료 합류 이벤트 시스템은 `EventBus.companion_join_requested(companion_id)` 신호 수신 한 번으로 동료 합류의 모든 결과를 올바른 순서로 실행하는 Feature 레이어 이벤트 핸들러다. 두 단계를 순서대로 처리한다: (1) **Phase A — 상태 갱신** (동기) — 해당 NPC의 `RelationshipState`를 `COMPANION`으로 전환하고 `PartyManager`에 동료를 등록, `companion_data_registered` 신호 발행(세이브·내부 시스템 즉시 반응용); (2) **Phase C — 청각+비주얼 연출** (await) — 동료 캐릭터 씬을 `CompanionLayer`에 인스턴스화한 직후 `play_sfx("companion_joined")` 재생, `join_in` 애니메이션 실행, 완료 후 `companion_appeared` 신호 발행(HUD 갱신용) 및 AI 활성화. HUD는 캐릭터가 화면에 등장한 시점에 갱신된다. 동료 영입 퀘스트 시스템이 유일한 호출자다. 플레이어는 이 시스템의 존재를 모르지만, 이 시스템이 없으면 Pillar 2(눈에 보이는 성장)의 가장 결정적인 순간 — 새 동료가 화면에 등장하는 순간 — 이 존재하지 않는다.

## Player Fantasy

화면에 사람 하나가 더 늘어나 있다. 방금까지 고개를 숙이고 서 있던 그 사람이 이제 내 옆에 나란히 서 있고, 그 짧은 연출음이 "끝났다"가 아니라 "이제 시작이다"라고 말해준다. 유랑단의 스노우볼은 레벨업 창이 아니라 이 한 컷 — 다음 전투에서 움직이는 실루엣이 하나 더 많다는 사실 — 에서 시작된다.

합류는 매번 똑같은 짧은 의식이다. 대화가 끝나고, 한 박자, 연출음, 그리고 새 동료가 내 옆에 서 있다. 세 번째 동료든 여덟 번째 동료든 이 리듬은 변하지 않고, 바로 그 반복이 "나의 유랑단이 또 한 명 커졌다"는 감각을 낡지 않은 채 지켜낸다.

## Detailed Design

### Core Rules

**R-1: 진입 조건 검증**
`EventBus.companion_join_requested(companion_id)` 수신 즉시 세 조건을 확인한다: (a) `NPCRegistry.get_npc(companion_id)` != null, (b) `PartyManager.is_recruited(companion_id)` == false, (c) `PartyManager.companion_count < MAX_PARTY_SIZE`. 조건 중 하나라도 실패 시 `push_warning()` + 즉시 중단. 이후 어떤 단계도 실행하지 않는다.

**R-2: 재진입 방지**
`_is_playing: bool` Guard Flag가 `true`인 동안 들어오는 `companion_join_requested` 신호는 무시 + `push_warning()`. 취소·큐잉 없음.

**R-3: Phase A — 상태 갱신 (동기, 순서 고정)**
R-1·R-2 통과 후 `_is_playing = true` 설정. 다음을 순서대로 즉시 실행:
1. `NPCRegistry.set_state(companion_id, RelationshipState.COMPANION)` — NPC 상태를 COMPANION(=3)으로 전환
2. `PartyManager.add_companion(companion_id)` — 파티 등록 및 `companion_count` 증가
3. `companion_data_registered(companion_id)` 신호 발행 — 세이브 시스템·퀘스트 오케스트레이터 등 **내부 시스템 전용**. HUD는 이 신호를 구독하지 않는다.

> **원자성 범위**: GDScript는 트랜잭션 롤백을 지원하지 않는다. "순서 고정"은 실행 순서만 보장하며 부분 실패 시 롤백은 없다. NPCRegistry 성공 후 PartyManager 실패 시 `push_error()` 후 `_is_playing = false` 복구 — Edge Cases EC-7 참조.

**R-4: Phase B → Phase C로 통합**
청각 연출(SFX)은 비주얼 연출 시작과 동시에 발생한다. `AudioManager.play_sfx("companion_joined")` 호출은 Phase C 씬 인스턴스화 직후로 이동됨 — R-5 참조. 별도 Phase B는 존재하지 않는다.

**R-5: Phase C — 청각+비주얼 연출 (await)**
다음을 순서대로 실행한다:

1. **CompanionLayer 조회**: `get_tree().get_first_node_in_group(&"companion_layer")`. 결과가 `null`이면 `push_error()` + `_is_playing = false` + 즉시 중단 (EC-8).
2. **companion_scenes 검증**: `companion_scenes.has(companion_id)` 확인. 누락 시 `push_error()` + `_is_playing = false` + 즉시 중단 (EC-4).
3. **씬 인스턴스화**: 씬을 `CompanionLayer`에 추가. 초기 위치: 플레이어 위치 + `COMPANION_SPAWN_OFFSET` (기본값 `Vector2(48, 16)`). `Hurtbox` 비활성화 (EC-5).
4. **SFX 재생** (fire-and-forget): `AudioManager.play_sfx("companion_joined")` 호출. `await` 없음. 오디오 실패는 경고 로그만 — 시퀀스 계속 진행.
5. **애니메이션 가드**: `animation_player.has_animation("join_in")` 확인. 누락 시 `push_error()` + `Hurtbox` 재활성화 + `companion_appeared` 발행 + `_is_playing = false` + 즉시 중단 (EC-9).
6. **애니메이션 실행**: `AnimationPlayer.play("join_in")` (알파 0→1, 0.4초). `await animation_player.animation_finished`.
7. **노드 생존 확인**: `await` 직후 `is_instance_valid(self)` 체크. `false`면 즉시 `return` (EC-1).
8. **완료 처리**: `Hurtbox` 재활성화. `companion_appeared(companion_id)` 신호 발행 — **HUD·파티 UI 갱신 전용**. `_is_playing = false`. 동료 AI FOLLOWING 상태 활성화.

> **companion_scenes 소유권**: `@export var companion_scenes: Dictionary[StringName, PackedScene]`. Godot 에디터 Inspector에서 각 `companion_id(StringName) → PackedScene` 매핑을 설정한다. 맵 로드 시 이미 인스턴스화된 CompanionJoinEvent 노드에 설정되어 있어야 하며, 모든 MVP 동료 씬이 등록된 상태가 게임 시작의 전제조건이다.

**R-6: 실행 순서 불변**
Phase A → Phase C 순서는 고정. Phase A의 NPCRegistry 갱신은 반드시 PartyManager 등록보다 먼저 — PartyManager가 NPC 데이터를 참조할 수 있기 때문. SFX는 Phase C 내부에서 씬 인스턴스화 직후 발행되며 별도 Phase를 구성하지 않는다.

### States and Transitions

상태 머신 불필요. `_is_playing` Guard Flag 하나로 재진입을 방지한다.

| 상태 | 설명 | `_is_playing` |
|------|------|---------------|
| 대기 | 신호 수신 가능 | `false` |
| 시퀀스 실행 중 | Phase A~C 진행 | `true` |

```
대기 ─ companion_join_requested ─► Phase A ─────────────► Phase C ─► 대기
 ▲                                                                       │
 └──────────────────── _is_playing = false ◄─── animation_finished ─────┘

Phase C 내부: 씬 인스턴스화 → SFX → 애니메이션 → companion_appeared
※ 시퀀스 실행 중 신호 수신 → push_warning + 무시
```

### Interactions with Other Systems

| 방향 | 시스템 | 인터페이스 | 타이밍 |
|------|--------|-----------|--------|
| ← 수신 | **EventBus** (동료 영입 퀘스트가 emit) | `companion_join_requested(companion_id)` | 트리거 |
| → 쓰기 | **NPCRegistry** | `set_state(id, COMPANION)` | Phase A, 1번째 |
| → 쓰기 | **PartyManager** | `add_companion(companion_id)` | Phase A, 2번째 |
| → 발행 | **내부 시스템** (세이브, 퀘스트 오케스트레이터) | `companion_data_registered(companion_id)` signal | Phase A 완료 직후 |
| → 씬 조작 | **CompanionLayer** (현재 맵, Group 조회) | 씬 인스턴스 추가 + 초기 위치 설정 | Phase C 진입 |
| → 호출 | **AudioManager** | `play_sfx("companion_joined")` | Phase C — 씬 인스턴스화 직후 |
| ← await | **AnimationPlayer** | `animation_finished` signal | Phase C 완료 감지 |
| → 발행 | **HUD / 파티 UI** | `companion_appeared(companion_id)` signal | Phase C 완료 직후 |

## Formulas

동료 합류 이벤트 시스템에는 수학적 공식(확률, 스케일링, 누적 계산)이 없다. 시스템의 모든 결과는 순서가 고정된 이벤트 시퀀스이며, 수치 연산은 타이밍 상수와 위치 오프셋 두 종류만 존재한다.

### F-1: 합류 애니메이션 지속 시간

```
JOIN_ANIMATION_DURATION = 0.4초
```

| 상수 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `JOIN_ANIMATION_DURATION` | float | [0.2, 1.0] 초 | `join_in` 애니메이션 전체 재생 시간. AnimationPlayer 리소스의 트랙 길이와 반드시 일치해야 한다. |

- **Worked example**: `JOIN_ANIMATION_DURATION = 0.4` → `join_in` 트랙 길이 0.4s, alpha keyframe: t=0.0 → 0.0, t=0.4 → 1.0
- **조율 메모**: 0.2s 미만은 너무 날카롭게 튀어오르고, 1.0s 초과는 연출이 늘어진다. 세 번째 동료부터 반복 경험되는 연출이므로 짧고 명확하게 유지한다.

### F-2: 동료 스폰 위치

```
spawn_position = player_position + COMPANION_SPAWN_OFFSET
```

| 심볼 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `player_position` | Vector2 | 맵 내 임의 좌표 | Phase C 시작 시점의 플레이어 월드 좌표 |
| `COMPANION_SPAWN_OFFSET` | Vector2 | x: [-64, 64], y: [-32, 32] | 플레이어 기준 동료 초기 위치 오프셋. 기본값 `Vector2(48, 16)`. y 오프셋 16px은 아이소메트릭 깊이감. |
| `spawn_position` | Vector2 | 맵 내 임의 좌표 | 동료 씬이 인스턴스화되는 월드 좌표 |

- **Worked example**: `player_position = Vector2(320, 240)`, `COMPANION_SPAWN_OFFSET = Vector2(48, 16)` → `spawn_position = Vector2(368, 256)`
- **맵 경계 클램프**: 이 시스템의 책임이 아니다. 맵-지역 시스템이 `CompanionLayer` 범위를 보장한다.

### F-3: companion_count 불변식 (Invariant)

공식 없음 — 상태 전이 규칙.

```
POST: companion_count == PRE: companion_count + 1
```

| 심볼 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `companion_count (PRE)` | int | [0, MAX_PARTY_SIZE - 1] | `add_companion()` 호출 전 파티 인원 수 |
| `companion_count (POST)` | int | [1, MAX_PARTY_SIZE] | `add_companion()` 호출 후 파티 인원 수 |

- **보장 책임**: `PartyManager.add_companion()` 내부. 이 시스템은 호출만 한다.
- **Worked example**: `companion_count = 2` → `add_companion("hunter")` → `companion_count = 3`

## Edge Cases

**EC-1: Phase C 실행 중 씬 전환**
`CompanionJoinEvent` 노드가 해제(freed)되면 `await animation_finished` 이후 코루틴 재개 시 `call_instance_null` 에러가 발생한다. 해결 (두 단계 모두 필요):
- `_exit_tree()` 콜백에서 `_is_playing = false` 강제 복구.
- Phase C의 `await animation_player.animation_finished` 직후 `is_instance_valid(self)` 체크 — `false`면 즉시 `return`.

**EC-2: Pause 중 await 정지**
`get_tree().paused = true` 상태에서 `AnimationPlayer`(기본 `PROCESS_MODE_PAUSABLE`)의 내부 클럭이 멈춰 `animation_finished`가 발행되지 않아 `_is_playing` 영구 잠금 위험. `CompanionJoinEvent` 노드의 `process_mode`만 `PROCESS_MODE_ALWAYS`로 설정하는 것은 **불충분** — `AnimationPlayer` 자체도 `PROCESS_MODE_ALWAYS`로 설정해야 Pause 중 애니메이션이 진행된다. 두 노드 모두 `PROCESS_MODE_ALWAYS` 설정 필수.

**EC-3: 시퀀스 실행 중 두 번째 합류 신호 수신**
R-2에 의해 무시 + `push_warning()`. 묵시적 유실이며 재시도 큐 없음. 두 번째 동료 영입이 완료된 상태에서 현재 시퀀스 종료(`_is_playing = false`) 후 퀘스트 오케스트레이터가 재시도 책임을 가져야 한다 — 동료 영입 퀘스트 GDD에 이 계약 명시 필요.

**EC-4: `companion_scenes[companion_id]` 누락**
Phase A·B는 이미 완료된 상태에서 Phase C 씬 로드 실패. NPCRegistry는 `COMPANION`, PartyManager는 등록 완료이나 화면에 동료 씬이 없는 "보이지 않는 동료" 상태가 된다. `push_error()` 후 `_is_playing = false` 복구. 맵에 해당 동료 없이 계속 진행 — 데이터는 일관성을 유지하나 비주얼 결함이 있음. MVP에서 잘못된 `companion_id` 전달은 개발자 실수로 간주.

**EC-5: 합류 애니메이션 중 즉시 피격**
`join_in` 재생 중(0.4초) `Hurtbox`가 활성화되면 동료가 등장하자마자 피해를 받는다. 해결: Phase C에서 씬 추가 직후 동료 루트 노드의 `Hurtbox` 를 비활성화하고 `animation_finished` 후 활성화. 단, 동료 AI가 FOLLOWING 상태로 전환되기 전에 발동하므로 즉사 처리는 동료 AI 시스템 R-6을 따른다.

**EC-6: 파티 최대 인원 초과 시도**
R-1의 `companion_count < MAX_PARTY_SIZE` 체크에서 이미 차단됨. `push_warning()` + 중단. `MAX_PARTY_SIZE` 값은 능력치 시스템 GDD에서 정의.

**EC-7: Phase A 부분 실패 — NPCRegistry 성공 후 PartyManager 실패**
`NPCRegistry.set_state()` 성공 후 `PartyManager.add_companion()`이 실패하면: NPC 상태는 `COMPANION`이나 파티 미등록 상태. `push_error()` 후 `_is_playing = false` 복구. 이후 `is_recruited()` 체크가 false를 반환해 재시도 가능 — 단, NPCRegistry 상태가 이미 COMPANION이므로 재진입 시 `set_state()`가 다시 호출됨. 이 불일치는 세이브/로드 시스템이 감지해야 한다.

**EC-8: `CompanionLayer` 참조 방식**
Phase C는 `CompanionLayer`를 매 실행 시점에 동적 조회한다 — 캐시 사용 금지. 씬 전환 후 새 맵의 `CompanionLayer`를 자동으로 참조하기 위함. **조회 방식**: `get_tree().get_first_node_in_group(&"companion_layer")` — 절대 경로(`get_node()`) 금지. 모든 맵 씬의 `CompanionLayer` 노드는 `companion_layer` 그룹에 속해야 한다 (씬 컨벤션 강제). 조회 결과가 `null`이면 `push_error()` + Phase C 중단. EC-4와 동일하게 Phase A 데이터는 유지.

**EC-9: `join_in` 애니메이션 미존재**
`AnimationPlayer.play("join_in")` 호출 전 `has_animation("join_in")` 체크 필수. 애니메이션이 없으면 Godot 4.x는 에러 로그 후 아무것도 하지 않아 `animation_finished`가 영원히 발행되지 않고 `_is_playing = true` 영구 잠금 상태가 된다. 해결: `has_animation()` 실패 시 `push_error()` + `Hurtbox` 재활성화 + `companion_appeared` 발행 + `_is_playing = false` 즉시 복구 후 중단. 씬이 이미 인스턴스화된 상태이므로 Phase A 데이터와 씬 모두 보존한 채 종료.

## Dependencies

**이 시스템이 의존하는 것 (Upstream)**

| 시스템 | 의존 내용 | 인터페이스 |
|--------|-----------|-----------|
| **동료 영입 퀘스트** | 유일한 호출자. `EventBus.companion_join_requested(companion_id)` emit | EventBus 신호 |
| **EventBus** | 신호 라우팅 인프라. Autoload Singleton | `companion_join_requested(companion_id: StringName)` |
| **NPCRegistry** | NPC 상태 갱신 | `set_state(id, COMPANION)`, `get_npc(id)` |
| **PartyManager** | 동료 등록, companion_count 관리 | `add_companion(id)`, `is_recruited(id)`, `companion_count` |
| **AudioManager** | 합류 연출음 재생 | `play_sfx("companion_joined")` |

**PartyManager 최소 계약 (이 GDD에서 인라인 정의 — 추후 별도 GDD 분리 가능)**
```
is_recruited(id: StringName) → bool    # 중복 합류 방지
add_companion(id: StringName) → void   # 등록 및 companion_count++ 
companion_count: int                   # 전투 GDD R-5 스케일링이 읽음
MAX_PARTY_SIZE: int                    # 최대 인원 (능력치 시스템 GDD에서 값 확정 예정)
```

**이 시스템에 의존하는 것 (Downstream)**

| 시스템 | 의존 내용 | 구독 신호 |
|--------|-----------|-----------|
| **실시간 파티 전투** | `PartyManager.companion_count` 증가 — 적 수 스케일링(전투 GDD R-5)에 즉시 반영 | (신호 불필요 — PartyManager 직접 읽기) |
| **동료 AI 시스템** | Phase C 완료 시 동료 씬 활성화, FOLLOWING 상태로 초기화 | `companion_appeared` |
| **HUD / 전투 UI** | 파티 표시 갱신 — 캐릭터가 화면에 나타난 시점에 갱신 | `companion_appeared` |
| **세이브/로드 시스템** | 파티 상태 변경 감지, 세이브 마킹 — Phase A 완료 직후 반응 | `companion_data_registered` |
| **퀘스트 오케스트레이터** | EC-3 재시도 계약: 두 번째 신호 드롭 감지 후 `_is_playing = false` 확인 후 재emit — **동료 영입 퀘스트 GDD에 이 계약 명시 필요** | `companion_data_registered` |

## Tuning Knobs

| 변수 | 기본값 | 안전 범위 | 영향 |
|------|--------|-----------|------|
| `JOIN_ANIMATION_DURATION` | 0.4초 | 0.2–1.0초 | 짧을수록 급격히 튀어오름, 길수록 연출 늘어짐. AnimationPlayer 리소스 트랙 길이와 반드시 동기화. |
| `COMPANION_SPAWN_OFFSET` | `Vector2(48, 16)` | x: [-64, 64], y: [-32, 32] | x 양수 = 플레이어 오른쪽 등장. y 양수 = 아이소메트릭 깊이. 절댓값 64 초과 시 너무 멀리 등장해 부자연스러움. |

> **조율 지침**: `JOIN_ANIMATION_DURATION`은 세 번째 동료부터 반복 경험되는 연출이다. 0.4초는 처음엔 인상적이고 열 번째도 리듬을 끊지 않는 균형점. 0.6s 이상은 연속 영입 시 답답하게 느껴진다.

## Visual/Audio Requirements

### 오디오

**`companion_joined` SFX**
- 성격: 따뜻하고 짧은 2음절 상승 톤 (벨+현악 레이어). "영입 성공"이 아니라 "새 동료, 새 출발"의 느낌.
- 길이: 0.8–1.2초 (애니메이션 0.4초보다 길어도 됨 — 여운이 남아야 함)
- 볼륨 카테고리: `SFX_UI` (BGM에 덮이지 않도록 별도 버스)
- 반복 내구성: 열 번 들어도 짧고 깔끔해야 함. 팡파르 금지.
- 구현 키: `AudioManager.play_sfx("companion_joined")`

### 비주얼

**`join_in` 애니메이션 스펙**
| 항목 | 값 | 비고 |
|------|----|------|
| 길이 | 0.4초 | `JOIN_ANIMATION_DURATION` 상수와 동기화 필수 |
| 트랙 | Alpha | t=0.0 → 0.0, t=0.4 → 1.0, Ease: Linear |
| 추가 트랙 (선택) | Scale | `Vector2(0.85, 0.85)` → `Vector2(1.0, 1.0)` — 살짝 커지며 등장. 과하면 제거. |
| 프레임 레이트 | 60fps 기준 | Compatibility Renderer 기준 |

**스폰 위치 시각 규칙**
- 플레이어 오른쪽 뒤(기본 `Vector2(48, 16)`) — 화면에서 "뒤에서 나타나는" 느낌
- 맵 타일 그리드(16px 또는 32px)에 스냅 권장 — 픽셀아트 정렬
- 플레이어 스프라이트와 겹치지 않아야 함 (x offset ≥ 32px 유지)

**기타 비주얼 규칙**
- 합류 연출 중 카메라 이동 없음 — 플레이어/맵 중심 유지
- 이펙트 파티클 없음 (MVP) — 연출은 fade-in + SFX만으로 완결
- 합류 후 첫 프레임: 동료 Idle 애니메이션 즉시 시작 (FOLLOWING 전환과 동시)

## Acceptance Criteria

| # | 조건 | 검증 방법 |
|---|------|-----------|
| AC-CJE-01 | **GIVEN** 진입 조건(NPCRegistry 등록, is_recruited false, companion_count < MAX) 충족 시 **WHEN** `companion_join_requested` emit **THEN** 콜 순서 레코더가 `[npc_registry_set_state, party_manager_add_companion, companion_data_registered, companion_scene_instantiate, play_sfx, animation_play, companion_appeared]` 순으로 기록되고, `companion_data_registered`는 반드시 `companion_appeared` 이전에 발행된다 | 단위 테스트 (GUT spy 패턴) |

| AC-CJE-03 | **GIVEN** `companion_count == MAX_PARTY_SIZE` **WHEN** 신호 emit **THEN** push_warning, 시퀀스 중단, NPCRegistry 상태 변경 없음 | 단위 테스트 |
| AC-CJE-04 | **GIVEN** 시퀀스 실행 중(`_is_playing == true`) **WHEN** 두 번째 신호 emit **THEN** push_warning, 두 번째 신호 무시, 첫 시퀀스 완료 후 자동 재시도 없음 | 단위 테스트 |
