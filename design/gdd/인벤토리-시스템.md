# 인벤토리 시스템 (Inventory System)

> **Status**: Done
> **Author**: Juwon + agents
> **Last Updated**: 2026-04-17
> **Implements Pillar**: Pillar 1 (모인 사람들) — 동료를 얻기 위한 퀘스트 아이템 수집의 기반

## Overview

인벤토리 시스템은 플레이어가 수집한 아이템의 보유 수량을 추적하는 Core 레이어 시스템이다. 아이템 데이터베이스에서 아이템 정의를 참조하고, 실제 보유 현황(`item_id → quantity`)을 관리한다. 퀘스트 상태 머신이 "이 아이템을 N개 가지고 있는가?"를 질의하는 것이 MVP에서의 핵심 사용처다.

MVP 범위를 명확히 제한한다: **장비 착용, 아이템 정렬, 드래그앤드롭 UI는 없다.** 인벤토리는 퀘스트 조건 확인을 위한 데이터 저장소다. UI는 Vertical Slice에서 추가.

**데이터 구조:**
```gdscript
# { item_id: String → quantity: int }
inventory: Dictionary = {}
```

## Player Fantasy

플레이어가 인벤토리 시스템을 직접 인식하지는 않는다. 하지만 "드디어 재료를 다 모았다"는 순간의 만족감은 이 시스템이 퀘스트 상태 머신에 정확한 수량을 전달했기에 가능하다.

## Detailed Rules

**R-1: 아이템 추가**
`add_item(item_id: String, amount: int = 1) -> void`
아이템 DB에 존재하는 `item_id`만 허용. 없으면 경고 로그 + 무시. `inventory[item_id] += amount`. `item_added(item_id, quantity)` 신호 발행.

**R-2: 아이템 제거**
`remove_item(item_id: String, amount: int = 1) -> bool`
보유량이 `amount` 이상이면 제거 후 `true` 반환. 부족하면 제거 없이 `false` 반환. `item_removed(item_id, quantity)` 신호 발행 (성공 시에만).

**R-3: 수량 조회**
`get_quantity(item_id: String) -> int`
보유 중이면 수량 반환. 없으면 0 반환.

**R-4: 보유 여부 조회**
`has_item(item_id: String, amount: int = 1) -> bool`
`get_quantity(item_id) >= amount` 반환. 퀘스트 상태 머신의 조건 확인 메서드.

**R-5: 용량 제한 없음**
MVP에서 인벤토리 칸 수 제한 없음. 플레이어는 무한히 수집 가능.

**R-6: Autoload Singleton**
`Inventory`로 등록. 씬 전환 시에도 보유 아이템 유지.

## Formulas

이 시스템에는 수학 공식이 없다. 모든 연산은 정수 덧셈/뺄셈이며 클램프 없음(R-5: 용량 무제한).

**F-1: 수량 갱신**
```
# 추가
inventory[item_id] = get_quantity(item_id) + amount

# 제거 (성공 시에만)
inventory[item_id] = get_quantity(item_id) - amount
if inventory[item_id] == 0:
    inventory.erase(item_id)   # 0인 항목은 딕셔너리에서 제거
```

**F-2: 조건 확인**
```
has_item(item_id, amount) = get_quantity(item_id) >= amount
```

> 수량이 0인 항목은 딕셔너리에서 제거해 메모리 낭비 방지. `get_quantity()`는 항목 없을 때 0 반환으로 통일.

## Edge Cases

**EC-1: 존재하지 않는 `item_id`로 `add_item()` 호출**
R-1에 정의됨 — 아이템 DB 조회 실패 시 경고 로그 + 무시. 인벤토리 변화 없음.

**EC-2: 보유량보다 많은 수량 `remove_item()` 요청**
R-2에 정의됨 — 제거 없이 `false` 반환. 부분 제거 없음. 퀘스트 상태 머신은 반환값을 확인해야 함.

**EC-3: `amount = 0`으로 `add_item()` / `remove_item()` 호출**
경고 로그 + 무시. 0 수량 연산은 의미 없음.

**EC-4: 같은 아이템을 여러 번 `add_item()`**
정상 동작. 수량 누적. 예: `add_item("herb", 3)` 두 번 → `get_quantity("herb") == 6`.

**EC-5: 씬 전환 후 인벤토리 유지**
R-6 Autoload로 보장. Singleton이 씬 전환에 걸쳐 살아남음.

**EC-6: 세이브/로드 시 인벤토리 직렬화**
`inventory` 딕셔너리는 JSON 직렬화 가능 (String 키, int 값). 세이브/로드 시스템이 `Inventory.inventory`를 직접 읽고 씀.

## Dependencies

**이 시스템이 의존하는 것 (Upstream)**
- **아이템 데이터베이스** — `item_id` 유효성 검증에 사용

**이 시스템에 의존하는 것 (Downstream)**

| 시스템 | 의존 내용 |
|--------|-----------|
| **퀘스트 상태 머신** | `has_item()`, `remove_item()`으로 퀘스트 조건 확인 및 아이템 소비 |
| **수집/자원 시스템** | `add_item()` 호출로 수집 아이템 인벤토리에 추가 |
| **세이브/로드 시스템** | `inventory` 딕셔너리 직접 읽기/쓰기 |

## Tuning Knobs

조정할 수치 없음. MVP에서 용량 제한 없고 아이템 수량은 퀘스트 설계로 결정. 아이템 정의(이름, 설명)는 아이템 데이터베이스에서 관리.

## Acceptance Criteria

| # | 조건 | 검증 방법 |
|---|------|-----------|
| AC-1 | `add_item("herb", 3)` 후 `get_quantity("herb") == 3` | 단위 테스트 |
| AC-2 | `has_item("herb", 3)` → `true`, `has_item("herb", 4)` → `false` | 단위 테스트 |
| AC-3 | `remove_item()` 성공 시 `true`, 수량 부족 시 `false` 반환 | 단위 테스트 |
| AC-4 | 수량이 0이 된 항목은 딕셔너리에서 제거된다 | 단위 테스트 |
| AC-5 | 존재하지 않는 `item_id`로 `add_item()` 호출 시 경고 로그만 출력되고 인벤토리 변화 없다 | 단위 테스트 |
| AC-6 | 씬 전환 후에도 인벤토리 보유 아이템이 유지된다 | 수동 플레이테스트 |
