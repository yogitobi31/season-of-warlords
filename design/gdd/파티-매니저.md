# 파티 매니저 (Party Manager)

> **Status**: Done
> **Author**: Juwon + agents
> **Last Updated**: 2026-04-26
> **Implements Pillar**: Pillar 2 (눈에 보이는 성장) — 합류한 동료의 수가 전투와 세계에 가시적으로 영향을 미침

## Overview

파티 매니저는 현재 파티에 합류한 동료 목록을 관리하는 Core 레이어 Autoload 싱글톤이다. 동료-합류-이벤트 시스템이 동료 합류를 완료한 후 파티에 등록하고, 실시간-파티-전투와 동료-AI가 파티 구성원 수와 목록을 읽는다. 씬 전환 시에도 파티 상태가 유지되어야 하므로 Autoload로 등록된다.

MVP에서 최대 파티 인원은 동료 3명이다 (`MAX_PARTY_SIZE = 3`). 플레이어는 파티 카운트에 포함되지 않는다.

## Player Fantasy

파티가 커질수록 게임이 달라진다는 것이 느껴져야 한다. 혼자 싸울 때와 동료 3명이 함께할 때의 차이는 수치가 아니라 화면의 밀도와 소리로 전달된다. 파티 매니저는 그 변화의 상태를 보관하는 보이지 않는 기반이다.

## Detailed Rules

**R-1: PartyManager Autoload**
`PartyManager`로 등록. 내부 상태: `_companions: Array[StringName]` — 현재 파티에 합류한 동료의 `companion_id` 목록. 씬 전환 시 유지.

**R-2: 동료 등록**
`register_companion(companion_id: StringName) -> void` — 동료-합류-이벤트 시스템이 합류 처리 완료 후 호출. 이미 등록된 `companion_id`면 무시. `MAX_PARTY_SIZE` 초과 시 등록 거부 + 경고 로그.

**R-3: 동료 조회**
```gdscript
var companion_count: int  # _companions.size() 읽기 전용 프로퍼티
func is_recruited(companion_id: StringName) -> bool
func get_companions() -> Array[StringName]  # 복사본 반환
```

**R-4: 최대 인원 상수**
```gdscript
const MAX_PARTY_SIZE: int = 3
```
동료-합류-이벤트 시스템이 합류 전 `companion_count < MAX_PARTY_SIZE` 확인. 파티 가득 찬 경우 합류 이벤트 자체가 발행되지 않아야 함 (동료-합류-이벤트 R-X에서 처리).

**R-5: 동료 이탈 (MVP 제외)**
MVP에서 동료가 파티를 이탈하는 시나리오 없음. `remove_companion()` 인터페이스는 정의하지 않음. Vertical Slice에서 필요 시 추가.

## Formulas

수학 공식 없음. 카운트 비교만 사용.
```
companion_count = _companions.size()          # 현재 파티 동료 수 (0–3)
can_join = companion_count < MAX_PARTY_SIZE   # 합류 가능 여부
```

| 상수 | 값 | 설명 |
|------|-----|------|
| `MAX_PARTY_SIZE` | 3 | 동료 최대 수. 플레이어 제외. |

## Edge Cases

**EC-1: 같은 동료 중복 등록**
`register_companion()` 호출 시 `_companions.has(companion_id)` 확인. 이미 있으면 경고 로그 + 무시.

**EC-2: MAX_PARTY_SIZE 초과 등록 시도**
경고 로그 + 등록 거부. 동료-합류-이벤트 시스템이 사전 확인을 통해 이 경로에 도달하지 않아야 함. 이중 방어.

**EC-3: 존재하지 않는 companion_id로 `is_recruited()` 호출**
`false` 반환. 오류 아님.

**EC-4: 씬 전환 중 파티 상태**
Autoload이므로 씬 전환에 영향 없음. `_companions` 배열은 게임 종료까지 유지.

## Dependencies

**이 시스템이 의존하는 것 (Upstream)**
- 없음. 순수 데이터 저장소. 다른 시스템의 신호나 데이터에 의존하지 않음.

**이 시스템에 의존하는 것 (Downstream)**

| 시스템 | 의존 내용 |
|--------|-----------|
| **동료-합류-이벤트** | `register_companion(companion_id)` 호출 — 합류 완료 시 파티 등록 |
| **동료-영입-퀘스트** | `is_recruited(companion_id)` — 중복 합류 방지 (EC-2) |
| **실시간-파티-전투** | `companion_count` — 적 스케일링 계산 (`enemy_count = base + companion_count × ENEMY_SCALE_PER_COMPANION`) |
| **세이브/로드 시스템** | `get_companions()` — 파티 상태 저장/복원 (Vertical Slice) |

**공개 인터페이스 요약**
```gdscript
const MAX_PARTY_SIZE: int = 3

var companion_count: int  # 읽기 전용

func register_companion(companion_id: StringName) -> void
func is_recruited(companion_id: StringName) -> bool
func get_companions() -> Array[StringName]
```

## Tuning Knobs

| 변수 | 기본값 | 안전 범위 | 영향 |
|------|--------|-----------|------|
| `MAX_PARTY_SIZE` | 3 | 1–5 | 늘리면 전투가 더 쉬워지고 화면이 복잡해짐. MVP 기준 3이 적절. |

## Acceptance Criteria

| # | 조건 | 검증 방법 |
|---|------|-----------|
| AC-1 | `register_companion(id)` 호출 후 `is_recruited(id)` 가 `true`를 반환한다 | 단위 테스트 |
| AC-2 | `companion_count`는 등록 수와 일치한다 | 단위 테스트 |
| AC-3 | 같은 id를 두 번 등록해도 `companion_count`는 1이다 | 단위 테스트 |
| AC-4 | `MAX_PARTY_SIZE` 초과 등록 시도 시 4번째 동료는 등록되지 않는다 | 단위 테스트 |
| AC-5 | `get_companions()` 반환값을 수정해도 내부 목록에 영향이 없다 | 단위 테스트 |
| AC-6 | 씬 전환 후에도 파티 목록이 유지된다 | 수동 플레이테스트 |
