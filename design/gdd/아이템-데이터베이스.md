# 아이템 데이터베이스 (Item Database)

> **Status**: In Revision (2026-04-25)
> **Author**: Juwon + Claude Code agents
> **Last Updated**: 2026-04-25
> **Implements Pillar**: Pillar 1 (Earned Fellowship) — QUEST_ITEM / is_quest_item RESOURCE 엔트리만 해당. CONSUMABLE은 전투 인프라 (Pillar 스폰서 없음).

## Overview

아이템 데이터베이스는 유랑단에 존재하는 모든 아이템의 정의를 저장하는 데이터 레이어다. 각 아이템의 이름, 설명, 카테고리(퀘스트 아이템 / 자원 / 소모품), 스택 가능 여부, 이 아이템이 연결된 동료/퀘스트 출처 등의 속성을 단일 `.tres` Resource로 정의하고, 인벤토리 시스템과 수집/자원 시스템에 단일 진실 출처로 제공한다. 플레이어는 이 시스템의 존재를 인식하지 않는다 — 대신 "이 약초를 3개 모아오면 진이 합류한다"는 퀘스트 조건이 의미를 가질 수 있는 것이 이 레이어 덕분이다. MVP에서 아이템은 장비가 아닌 퀘스트 조건 충족과 자원 수집에 집중한다 — 복잡한 장비/스킬 시스템은 MVP 범위 밖이다.

## Player Fantasy

플레이어는 아이템 데이터베이스를 인식하지 않는다. 이 시스템이 제대로 작동할 때 플레이어가 느끼는 것: 인벤토리를 열었을 때 "낡은 열쇠"와 "약초"가 각자 다른 사연을 가진 것처럼 느껴지는 것. 퀘스트 조건 아이템을 손에 넣었을 때 "이걸 가져다주면 저 사람이 합류하는구나" 하는 목적 있는 소지감. 아이템이 그냥 숫자가 아니라 누군가의 이야기와 연결된 물건처럼 느껴지는 것 — 그것이 Pillar 1 (Earned Fellowship)의 물질적 기반이다.

이 시스템이 실패하면 플레이어가 느끼는 것: 모든 아이템이 똑같이 무미건조하게 보인다. 퀘스트 조건 아이템과 일반 자원이 구분되지 않아 인벤토리가 어수선하다. "이게 뭐였지?"라는 혼란 — Pillar 1이 물질적 근거를 잃는다.

## Detailed Rules

### ItemDefinition 필드 정의

모든 `ItemDefinition` Resource 인스턴스는 다음 필드를 가진다. 모든 필드는 `@export`로 선언하여 `.tres` 직렬화와 인스펙터 편집을 보장한다.

| 필드 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `item_id` | `StringName` | — | 아이템 고유 키. 초기화 후 변경 금지. ItemDB Dictionary 키와 일치해야 함 |
| `display_name` | `String` | `""` | UI 표시명 |
| `description` | `String` | `""` | 인벤토리 툴팁 |
| `icon` | `Texture2D` | `null` | 인벤토리 아이콘. `null`이면 등록 시 `push_error` 발생 |
| `category` | `int` | `RESOURCE(1)` | ItemCategory enum 값 |
| `is_quest_item` | `bool` | `false` | 퀘스트 조건 체크 fast-path. `QUEST_ITEM` 카테고리이면 초기화 시 자동 `true` |
| `stackable` | `bool` | `true` | 스택 가능 여부 |
| `max_stack` | `int` | `99`¹ | 최대 스택 수. `1` 이상이어야 함. `stackable=false`이면 `1`로 강제. ¹CONSUMABLE 카테고리는 `_validate_definitions()` 시 `CONSUMABLE_MAX_STACK_DEFAULT(20)`로 보정 |
| `quest_tags` | `Array[StringName]` | `[]` | 이 아이템이 조건인 퀘스트 ID 목록 |
| `source_companion_id` | `StringName` | `&""` | 이 아이템이 연결된 동료 ID. 비어 있으면 특정 동료와 무관한 일반 아이템 |
| `source_quest_id` | `StringName` | `&""` | 이 아이템이 연결된 퀘스트 ID. 비어 있으면 일반 아이템 |
| `effect_type` | `int` | `NONE(0)` | EffectType enum. CONSUMABLE 전용. 비-CONSUMABLE 아이템은 `NONE` |
| `effect_value` | `float` | `0.0` | 효과 수치. HEAL이면 회복량. 비-CONSUMABLE은 `0.0` |

**Enum 정의:**

```
ItemCategory: QUEST_ITEM=0, RESOURCE=1, CONSUMABLE=2
EffectType: NONE=0, HEAL=1  (MVP에서 HEAL만 구현. 이후 확장 예정)
```

**카테고리별 상수:**

```
CONSUMABLE_MAX_STACK_DEFAULT: int = 20
```

`_validate_definitions()` 실행 시 `category == CONSUMABLE AND max_stack == 99`(필드 기본값 센티넬)이면 `CONSUMABLE_MAX_STACK_DEFAULT`로 보정한다. 이것은 "잘못된 값 자동 수정 금지" 원칙과 충돌하지 않는다 — 잘못된 값이 아니라 "범주별 기본값 미설정" 상태를 채우는 것이기 때문이다.

**@export 타입 지침 (Godot 4.6):**
`category`와 `effect_type`은 `int`가 아닌 enum 타입으로 선언한다. Godot 4.6은 `@export var category: ItemCategory`를 직접 지원하며, `.tres` 직렬화 시 정수로 저장되므로 세이브 파일 호환성에 영향 없다. 인스펙터 드롭다운과 컴파일러 범위 검사를 얻기 위해 enum 타입 사용.

```
@export var category: ItemCategory = ItemCategory.RESOURCE
@export var effect_type: EffectType = EffectType.NONE
```

**`Array[StringName]` @export 직렬화 주의사항 (Godot 4.6):**
`quest_tags`는 `@export var quest_tags: Array[StringName]`으로 선언한다. 인스펙터에서 직접 편집 시 각 원소를 `&"quest_id"` 형식(StringName 리터럴)으로 입력해야 한다. `"quest_id"` (String 리터럴)로 입력하면 역직렬화 시 String으로 로드되어 `quest_tags.has()` 비교가 조용히 실패할 수 있다.

**Autoload 로드 전략 (Godot 4.6):**
ItemDB는 `_ready()` 진입 전 `_init()`에서 빈 Dictionary를 초기화하고, `.tres` 파일 로드는 `_ready()`에서 수행한다. 로드 메커니즘: 지정된 디렉토리(`res://assets/data/items/`)를 `DirAccess`로 스캔하여 모든 `.tres` 파일을 로드한다. Project Settings Autoload 등록 순서에서 ItemDB는 인벤토리 시스템, 퀘스트 상태 머신, 수집/자원 시스템보다 앞에 위치해야 한다.

### Core Rules

1. **모든 아이템은 하나의 `ItemDefinition` Resource로 정의된다.** 퀘스트 아이템, 자원, 소모품 구조 차이 없음. 차이는 `category`, `is_quest_item`, `effect_type` 값으로 표현한다.

2. **MVP 아이템 카테고리는 3개다:**

| 값 | 상수명 | 의미 | 스택 | 효과 필드 |
|----|--------|------|------|-----------|
| 0 | `QUEST_ITEM` | **고유 소지 아이템 전용** (열쇠, 편지, 특수 물건). 단일 수량 보유가 의미 있는 아이템. 퀘스트 완료 시 자동 제거 | `stackable=false, max_stack=1` 권장 | `NONE` |
| 1 | `RESOURCE` | 전투/탐험 중 획득하는 수집 자원. **수량 수집이 필요한 퀘스트 조건 아이템도 이 카테고리 사용.** | 가능 (기본 max 99) | `NONE` |
| 2 | `CONSUMABLE` | 사용 시 즉시 효과. MVP에서 `HEAL` 타입만 지원 | 가능 (기본 max 20) | `effect_type`, `effect_value` |

   **카테고리 사용 규칙:**
   - "약초 3개 모아오기" 같은 수집형 조건 아이템 → **RESOURCE** + `is_quest_item=true` + `quest_tags` 설정
   - "낡은 열쇠" 같은 특수 고유 아이템 → **QUEST_ITEM**
   - 장비(`EQUIPMENT`) 카테고리는 MVP 범위 밖

3. **`ItemDefinition`의 `@export` 필드는 직렬화 친화적 타입만 사용한다.** `Texture2D` 참조는 uid:// 기반으로 저장된다. 에셋 경로 변경 시 Godot의 UID 시스템이 링크를 유지하지만, 초기화 시 `icon == null` 검증을 통해 누락된 아이콘을 조기에 감지한다.

4. **`ItemDB`는 Autoload 싱글톤이다.** `ItemDB.get(item_id)` 단일 경로로 인벤토리 시스템, 수집/자원 시스템, 퀘스트 상태 머신이 접근한다. NPCRegistry와 동일한 Autoload 예외 근거 — ADR로 통합 문서화한다.

5. **아이템 정의는 에디터에서만 추가한다.** 런타임 중 새 ItemDefinition 생성 금지. ItemDB에 등록된 `.tres` 파일만 유효하다.

6. **퀘스트 아이템 조건 체크는 Push(signal) 방식이다.** 인벤토리 시스템이 `item_count_changed(item_id, new_count)` signal을 emit하면, 퀘스트 상태 머신이 `is_quest_item`을 먼저 확인해 비퀘스트 아이템을 즉시 필터링하고, `quest_tags`로 관련 퀘스트만 체크한다.

7. **퀘스트 조건 자원 사전 수집 정책 (MVP):** 퀘스트 수락 전에 보유한 RESOURCE 카테고리 퀘스트 조건 아이템도 퀘스트 조건으로 인정한다. 드롭 게이팅(퀘스트 수락 전 미드롭 등) 구현 여부는 수집/자원 시스템 GDD에서 결정.

8. **에러/경고 리포팅은 인젝터블 훅으로 제공한다.** `push_error()`/`push_warning()` 직접 호출 외에, GUT 단위 테스트에서 캡처 가능한 `error_occurred(message: String)` 신호를 ItemDB에 추가한다. NPCRegistry와 동일한 패턴 — 두 시스템 공통 설계로 ADR에 문서화.

### 자동 보정 규칙 (init-time validation)

`_validate_definitions()` 실행 시 다음 불일치는 **등록 거부 + push_error**로 처리한다. 자동 수정하지 않는다 — 데이터 작성자가 `.tres`를 직접 수정해야 한다. 이는 공유 Resource 변경(Godot 4 Resources는 같은 경로를 로드하면 동일 객체 반환)으로 인한 디스크 오염을 방지한다.

| 조건 | 처리 |
|------|------|
| `category == QUEST_ITEM` AND `is_quest_item == false` | push_error + 등록 거부 |
| `stackable == false` AND `max_stack != 1` | push_error + 등록 거부 |
| `max_stack < 1` | push_error + 등록 거부 |
| `icon == null` | push_error + 등록 거부 |
| 중복 `item_id` | `assert(false)` (디버그) + push_error (릴리즈) + 등록 거부 |
| `is_quest_item == true` AND `quest_tags.is_empty()` | push_error + 등록 거부 |
| `category != CONSUMABLE` AND `effect_type != NONE` | push_warning (등록은 허용, 설계 오류 경고) |
| `category == CONSUMABLE` AND `max_stack == 99` (필드 기본값 센티넬) | `max_stack = CONSUMABLE_MAX_STACK_DEFAULT(20)`으로 보정 후 등록 (자동 수정 아님 — 범주별 기본값 채우기) |
| `category == CONSUMABLE` AND `effect_type != NONE` AND `effect_value <= 0.0` | push_error + 등록 거부 (제로 또는 음수 효과 소모품은 명백한 데이터 오류) |
| `quest_tags.is_not_empty()` AND `is_quest_item == false` | push_error + 등록 거부 (quest_tags가 있는데 is_quest_item=false — Rule 6의 미러 케이스, 퀘스트 fast-path가 이 아이템을 건너뜀) |

### States and Transitions

아이템 데이터베이스는 정적 정의 레이어다. `ItemDefinition` 자체는 변경되지 않는다. 아이템 인스턴스(수량)의 생명주기:

| 단계 | 설명 | 소유자 |
|------|------|--------|
| **정의 (Definition)** | `.tres` 파일로 저장된 불변 아이템 스펙 | ItemDB (에디터) |
| **보유 (Held)** | 인벤토리에서 수량이 1 이상인 상태 | 인벤토리 시스템 |
| **소비 (Consumed)** | 사용 또는 퀘스트 완료로 제거된 상태 | 인벤토리 시스템 / 퀘스트 상태 머신 |

### Interactions with Other Systems

| 방향 | 시스템 | 읽는/쓰는 값 | 인터페이스 |
|------|--------|-------------|-----------|
| **→ (출력)** | 인벤토리 시스템 | `item_id`, `display_name`, `stackable`, `max_stack`, `icon`, `source_companion_id`, `source_quest_id` | `ItemDB.get(item_id)` |
| **→ (출력)** | 수집/자원 시스템 | `item_id`, `category`, `description` | `ItemDB.get(item_id)` |
| **→ (출력)** | 퀘스트 상태 머신 | `is_quest_item`, `quest_tags` | `ItemDB.get(item_id)` — signal 수신 후 조회 |
| **← (입력)** | 없음 — 읽기 전용 레이어 | — | — |

⚠️ **잠정적 인터페이스**: 인벤토리 시스템과 수집/자원 시스템 GDD 작성 시 위 계약 검증 필요.

### 교차-시스템 인터페이스 계약 (Cross-System Contracts)

**[계약 1] QUEST_ITEM 자동 제거 — 퀘스트 상태 머신 책임:**
퀘스트 상태 머신은 퀘스트 완료(`on_complete`) 시 다음 인터페이스를 호출해야 한다:
```
InventorySystem.remove_item(item_id: StringName, quantity: int) → bool
```
이 호출은 퀘스트 상태 머신 GDD의 `on_complete` 블록에 명시되어야 한다. 현재 퀘스트 상태 머신 GDD에 아이템 제거 단계가 없다 — **퀘스트 상태 머신 GDD 재검토 필요 (Producer 조정).**

**[계약 2] 인벤토리 시스템 효과 적용 전 카테고리 검사 의무:**
인벤토리 시스템이 아이템 사용 효과를 적용할 때는 반드시 `category == CONSUMABLE`을 먼저 검사해야 한다. 비-CONSUMABLE 아이템에 `effect_type != NONE`이 설정된 경우(Rule 7: push_warning 등록 허용) 효과를 무시해야 한다.
```
if item_def.category == ItemCategory.CONSUMABLE and item_def.effect_type != EffectType.NONE:
    apply_effect(item_def.effect_type, item_def.effect_value)
```

**[계약 3] 직접 .tres 로드 금지:**
다운스트림 시스템은 `ItemDefinition` 파일을 직접 `load()` / `preload()`로 로드해서는 안 된다. 모든 아이템 정의 접근은 `ItemDB.get(item_id)` 또는 `ItemDB.has(item_id)`를 통해야 한다. 직접 로드 시 `_validate_definitions()`가 실행되지 않아 max_stack 보정 등이 적용되지 않는다.

**[계약 4] Player Fantasy 전달 — 인벤토리 UI 시각적 차별화 의무:**
인벤토리 UI는 `source_companion_id`가 비어 있지 않은 아이템을 시각적으로 구분해야 한다 (테두리 색상, 아이콘 오버레이, 툴팁 배지 — 구체적 처리는 UX 명세에서 결정). 이 차별화가 없으면 QUEST_ITEM/is_quest_item RESOURCE의 Pillar 1 (Earned Fellowship) 판타지가 전달되지 않는다. `is_quest_item == true` 아이템도 동일한 시각적 처리를 받아야 한다.

## Formulas

이 시스템은 수치 계산 공식을 포함하지 않는다. 쿼리 인터페이스:

**Query 1 — 단일 아이템 정의 조회**

`definition = ItemDB.get(item_id: StringName) → ItemDefinition | null`

| Parameter | Type | Description |
|-----------|------|-------------|
| `item_id` | StringName | 아이템 고유 키 |
| 반환값 | ItemDefinition \| null | 등록된 아이템이면 정의 반환, 없으면 null + `error_occurred` 신호 |

**Query 2 — 아이템 존재 확인**

`exists = ItemDB.has(item_id: StringName) → bool`

`has()`는 `error_occurred`를 발생시키지 않는다 — 존재 여부를 묻는 것은 합법적 조회.

**Query 3 — 퀘스트 태그 역조회 (옵션, VS+)**

`items = ItemDB.get_by_quest_tag(quest_id: StringName) → Array[ItemDefinition]`

MVP에서는 불필요. VS+ 단계에서 퀘스트 에디터 UI가 생길 경우 유용.

**Output Range**: MVP 기준 7~14개 ItemDefinition 정의. `get()`은 O(1) Dictionary 조회.

## Edge Cases

**데이터 무결성 케이스:**

- **If `ItemDB.get(item_id)`가 null을 반환하는 경우**: null 반환 + `error_occurred` 신호. 호출 시스템이 null 체크를 해야 한다. 미등록 아이템 참조는 항상 개발 오류.

- **If 등록 검증 실패 조건을 가진 아이템이 등록되는 경우** (위 자동 보정 규칙 표 참조): 등록 거부 + `error_occurred` 신호. 해당 아이템은 `ItemDB.has()`에서 `false`를 반환한다.

- **If `max_stack < 1`인 경우** (stackable=true이더라도): 등록 거부 + `error_occurred`. 인벤토리 시스템이 슬롯 계산 시 `max_stack`으로 나누기를 수행할 때 0 또는 음수가 들어가면 런타임 오류가 발생한다.

**아이템 등록 케이스:**

- **If 같은 `item_id`로 두 개의 `.tres`가 등록된 경우**: `assert(false)` (디버그 빌드에서 즉시 중단) + push_error (릴리즈 빌드) + 두 번째 등록 거부. 첫 번째 정의를 유지한다.

- **If 게임 업데이트로 아이템이 삭제되었는데 구버전 세이브에 해당 item_id가 남아 있는 경우**: 인벤토리 시스템이 `ItemDB.has(item_id)`로 체크 후 `false`이면 해당 슬롯을 처리. 구체적 처리 방식은 인벤토리 GDD에서 결정.

**퀘스트 연동 케이스:**

- **If `QUEST_ITEM`이 퀘스트 완료 후 자동 제거되어야 하는데 인벤토리에 남아 있는 경우**: 제거 책임은 퀘스트 상태 머신에 있다. 제거 타이밍과 조건은 퀘스트 GDD에서 정의.

- **If `RESOURCE` 카테고리 아이템이 퀘스트 조건으로도 쓰이는 경우** (예: 약초가 자원이면서 영입 퀘스트 조건): `is_quest_item=true`, `quest_tags`에 해당 퀘스트 ID 추가. 카테고리는 `RESOURCE` 유지. 퀘스트 완료 후 요구 수량만 제거하고 나머지는 인벤토리에 남는다. "남은 수량 제거 vs 요구 수량만 제거" 구체적 처리는 퀘스트 상태 머신 GDD에서 결정.

- **If 동일 RESOURCE가 복수 퀘스트 조건으로 동시 사용되는 경우** (Quest A도 약초 3개 필요, Quest B도 약초 3개 필요, 현재 3개 보유): 두 퀘스트가 동시에 완료 조건을 충족한다. 소비 순서와 충돌 해소는 퀘스트 상태 머신 GDD에서 정의해야 한다 — "먼저 완료된 퀘스트가 소비"가 기본 정책 권장이지만 이 시스템의 책임 범위 밖.

**CONSUMABLE 케이스:**

- **If `effect_type`이 `NONE`인 CONSUMABLE이 사용된 경우**: 효과가 없는 소모품. 인벤토리 시스템은 수량을 1 감소시키지만 효과를 적용하지 않는다. 개발 중 플레이스홀더로 허용, 릴리즈 빌드에서는 push_warning으로 경고.

- **If 비-CONSUMABLE 아이템의 `effect_type`이 `NONE`이 아닌 경우**: 등록 시 push_warning. 수치는 저장되지만 인벤토리 시스템이 이를 무시해야 한다.

## Dependencies

**이 시스템이 의존하는 것 (Upstream):**

없음. 아이템 데이터베이스는 Foundation 레이어.

**이 시스템에 의존하는 것 (Downstream):**

| 시스템 | 의존 유형 | 읽는 값 | 의존 강도 |
|--------|-----------|---------|---------|
| 인벤토리 시스템 | Hard | `item_id`, `display_name`, `stackable`, `max_stack`, `icon`, `source_companion_id`, `source_quest_id`, `effect_type`, `effect_value` | 없으면 아이템 표시/스택/효과 규칙 없음 |

## Visual/Audio Requirements

이 시스템은 직접 렌더링하지 않지만, 다운스트림 UI 시스템에 다음 요구사항을 위임한다.

**아이콘 (Visual):**
- 모든 아이템은 유효한 `icon: Texture2D`를 가져야 한다 (null이면 등록 거부).
- 아이콘 크기와 팔레트는 아트 바이블(`design/art/art-bible.md`)을 따른다.

**시각적 차별화 (Player Fantasy 전달 계약):**
인벤토리 UI 시스템은 다음 두 조건에 해당하는 아이템을 플레이어에게 시각적으로 구분해 표시해야 한다:
- `is_quest_item == true` 또는 `source_companion_id != &""`인 아이템 → 인벤토리 슬롯에 구분 처리 필요

구체적 구현 방식 (테두리 색상 / 아이콘 오버레이 / 툴팁 배지 중 선택)은 UX 설계 단계에서 결정한다.

**오디오:**
아이템 획득 음향은 수집/자원 시스템 GDD에서 정의한다. 이 시스템은 오디오를 직접 트리거하지 않는다.

## UI Requirements

이 시스템은 직접 UI를 렌더링하지 않지만, 인벤토리 UI에 다음 데이터 계약을 제공한다.

**툴팁 표시 최소 요구사항:**
- `display_name`: 아이템 이름
- `description`: 설명 텍스트
- `source_companion_id`가 비어 있지 않으면: "[ 동료명 ]의 영입 조건" 표시 (동료 이름 조회는 NPCRegistry 경유)
- `source_quest_id`가 비어 있지 않으면: 퀘스트 연관 표시 (퀘스트 이름 조회는 퀘스트 상태 머신 GDD에서 결정)

**아이템 카테고리 구분 표시:**
- `QUEST_ITEM` 카테고리: 고유 아이템 마킹 필요 (단순히 수량이 아닌 '사연' 있는 물건임을 전달)
- `RESOURCE + is_quest_item=true`: 퀘스트 조건 자원임을 인식할 수 있도록 표시

구체적인 UX 구현은 `/ux-design` 명세에서 결정한다.

## Acceptance Criteria

### Core Rule 조건

**AC-ITEM-CR-01** (단일 아이템 조회)
- **GIVEN** ItemDB에 `item_id = &"herb_moonleaf"`인 아이템이 등록되어 있고
- **WHEN** `ItemDB.get(&"herb_moonleaf")`를 호출하면
- **THEN** 반환값이 null이 아니고, `item_id == &"herb_moonleaf"`인 `ItemDefinition`이어야 한다

**AC-ITEM-CR-02** (미등록 아이템 조회 — null 반환)
- **GIVEN** ItemDB에 등록되지 않은 item_id가 있을 때
- **WHEN** `ItemDB.get(&"nonexistent")`를 호출하면
- **THEN** `null`을 반환해야 한다

**AC-ITEM-CR-02b** (미등록 아이템 조회 — error_occurred 발생)
- **GIVEN** ItemDB에 `error_occurred` 신호 수신자가 연결되어 있고
- **WHEN** `ItemDB.get(&"nonexistent")`를 호출하면
- **THEN** `error_occurred` 신호가 1회 발생해야 한다

**AC-ITEM-CR-03** (`has()` 동작 — 에러 없음)
- **GIVEN** ItemDB에 `&"old_key"`가 등록되어 있고 `&"missing_item"`은 없을 때
- **WHEN** 각각 `ItemDB.has()`를 호출하면
- **THEN** `has(&"old_key")`는 `true`, `has(&"missing_item")`는 `false`를 반환해야 하며, `error_occurred` 신호는 발생하지 않아야 한다

**AC-ITEM-CR-04** (카테고리 3개 존재, 장비 없음)
- **GIVEN** `ItemCategory` 열거형이 정의되어 있고
- **WHEN** 상수 목록을 검사하면
- **THEN** `QUEST_ITEM(0)`, `RESOURCE(1)`, `CONSUMABLE(2)` 3개가 존재해야 하고, `EQUIPMENT` 상수가 없어야 한다

**AC-ITEM-CR-05** (QUEST_ITEM + is_quest_item=false — 등록 거부)
- **GIVEN** `category=QUEST_ITEM, is_quest_item=false`로 설정된 ItemDefinition이 있고
- **WHEN** ItemDB 초기화가 실행되면
- **THEN** `error_occurred` 신호가 발생하고, 해당 아이템이 ItemDB에 등록되지 않아야 한다 (`has(item_id) == false`)

**AC-ITEM-CR-06** (stackable=false + max_stack>1 — 등록 거부)
- **GIVEN** `stackable=false, max_stack=10`으로 설정된 ItemDefinition이 있고
- **WHEN** ItemDB 초기화가 실행되면
- **THEN** `error_occurred` 신호가 발생하고, 해당 아이템이 ItemDB에 등록되지 않아야 한다

**AC-ITEM-CR-07** (중복 item_id — 등록 거부)
- **GIVEN** 동일한 `item_id`를 가진 두 번째 ItemDefinition이 ItemDB에 등록 시도될 때
- **WHEN** ItemDB 초기화가 실행되면
- **THEN** `error_occurred` 신호가 발생하고, 두 번째 정의가 등록되지 않으며 첫 번째 정의가 유지되어야 한다

**AC-ITEM-CR-08** (max_stack=0 — 등록 거부)
- **GIVEN** `stackable=true, max_stack=0`으로 설정된 ItemDefinition이 있고
- **WHEN** ItemDB 초기화가 실행되면
- **THEN** `error_occurred` 신호가 발생하고, 해당 아이템이 ItemDB에 등록되지 않아야 한다

**AC-ITEM-CR-09** (icon=null — 등록 거부)
- **GIVEN** `icon=null`인 ItemDefinition이 있고
- **WHEN** ItemDB 초기화가 실행되면
- **THEN** `error_occurred` 신호가 발생하고, 해당 아이템이 ItemDB에 등록되지 않아야 한다

**AC-ITEM-CR-10** (CONSUMABLE effect fields)
- **GIVEN** `category=CONSUMABLE, effect_type=HEAL(1), effect_value=50.0`인 ItemDefinition이 등록되어 있고
- **WHEN** `ItemDB.get(item_id)`를 호출하면
- **THEN** `effect_type == 1`이고 `effect_value == 50.0`이어야 한다

**AC-ITEM-CR-11** (CONSUMABLE max_stack 범주별 기본값 보정)
- **GIVEN** `category=CONSUMABLE`인 ItemDefinition이 `.tres`에 `max_stack` 필드 없이 (GDScript 기본값 99가 적용된 상태로) 등록될 때
- **WHEN** `_validate_definitions()` 완료 후 `ItemDB.get(item_id).max_stack`을 읽으면
- **THEN** `ItemDB.CONSUMABLE_MAX_STACK_DEFAULT`(현재 `20`)를 반환해야 한다 — `_validate_definitions()`에서 센티넬 값 99를 범주별 기본값으로 보정했음을 확인

**AC-ITEM-CR-12** (source 필드 조회)
- **GIVEN** `source_companion_id=&"companion_jin", source_quest_id=&"quest_jin_herb"`인 아이템이 등록되어 있고
- **WHEN** `ItemDB.get(item_id)`를 호출하면
- **THEN** `source_companion_id == &"companion_jin"` 및 `source_quest_id == &"quest_jin_herb"`여야 한다

### Edge Case 조건

**AC-ITEM-CR-13** (비-CONSUMABLE + effect_type!=NONE — 경고 발생하나 등록 허용)
- **GIVEN** `category=RESOURCE, effect_type=HEAL(1)`인 ItemDefinition이 있고
- **WHEN** `_validate_definitions()`가 실행되면
- **THEN** push_warning이 발생하고, 해당 아이템은 **등록되어야** 한다 (`has(item_id) == true`) — Rule 7의 경고-허용 경로 확인

---

**AC-ITEM-EC-01** (quest_tags 비어 있는 is_quest_item=true — 등록 거부)
- **GIVEN** `is_quest_item=true`이고 `quest_tags`가 비어 있는 ItemDefinition이 등록 시도될 때
- **WHEN** ItemDB `_validate_definitions()`가 실행되면
- **THEN** `error_occurred` 신호가 발생하고, 해당 아이템이 등록되지 않아야 한다

**AC-ITEM-EC-02** (퀘스트 fast-path — is_quest_item=false 아이템)
- **GIVEN** `is_quest_item=false`인 `RESOURCE` 카테고리 아이템의 `item_count_changed` signal이 발생하고
- **WHEN** 퀘스트 상태 머신의 signal 핸들러가 실행되면
- **THEN** `is_quest_item` 체크 이후 퀘스트 조건 평가 로직에 진입하지 않아야 한다
- **[DEFERRED — 퀘스트 상태 머신 GDD 완성 후 Integration 테스트로 전환]** 퀘스트 상태 머신 GDD의 signal 핸들러 명세 완료 시 이 GDD와 퀘스트 상태 머신 GDD 양쪽에 교차 참조 Integration AC로 등록.

**AC-ITEM-EC-04** (CONSUMABLE + HEAL + effect_value=0.0 — 등록 거부)
- **GIVEN** `category=CONSUMABLE, effect_type=HEAL(1), effect_value=0.0`인 ItemDefinition이 있고
- **WHEN** `_validate_definitions()`가 실행되면
- **THEN** `error_occurred` 신호가 발생하고, 해당 아이템이 ItemDB에 등록되지 않아야 한다 (`has(item_id) == false`) — 제로 회복 소모품은 데이터 오류로 처리

**AC-ITEM-EC-05** (quest_tags 비어 있지 않음 + is_quest_item=false — 등록 거부)
- **GIVEN** `is_quest_item=false`이고 `quest_tags=[&"quest_jin_herb"]`(비어 있지 않음)인 ItemDefinition이 있고
- **WHEN** `_validate_definitions()`가 실행되면
- **THEN** `error_occurred` 신호가 발생하고, 해당 아이템이 등록되지 않아야 한다 — Rule 6의 미러 케이스: quest_tags가 있으면 is_quest_item도 true여야 함

**AC-ITEM-EC-06** (RESOURCE + is_quest_item=true — QUEST_ITEM 거부 규칙과 무관하게 등록 허용)
- **GIVEN** `category=RESOURCE, is_quest_item=true, quest_tags=[&"quest_jin_herb"]`인 ItemDefinition이 있고
- **WHEN** `_validate_definitions()`가 실행되면
- **THEN** `error_occurred` 신호가 발생하지 않고, 해당 아이템이 등록되어야 한다 (`has(item_id) == true`) — RESOURCE + is_quest_item=true는 유효한 조합이며 AC-ITEM-CR-05(QUEST_ITEM 카테고리 불일치 거부)와 무관함을 확인

**AC-ITEM-EC-03** (RESOURCE + quest_tags 양방향 경로)
- **GIVEN** `category=RESOURCE, is_quest_item=true, quest_tags=[&"quest_jin_herb"]`인 아이템이 등록되어 있고
- **WHEN** `ItemDB.get(item_id)`를 호출하면
- **THEN** `category == RESOURCE`이고, `is_quest_item == true`이며, `quest_tags.has(&"quest_jin_herb") == true`여야 한다
