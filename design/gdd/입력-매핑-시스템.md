# 입력 매핑 시스템 (Input Mapping System)

> **Status**: Designed (pending /design-review)
> **Author**: Juwon + agents
> **Last Updated**: 2026-04-17
> **Implements Pillar**: 모든 Pillar의 실행 기반 인프라

## Overview

입력 매핑 시스템은 게임 내 모든 액션 이름과 그 입력 바인딩을 관리하는 Foundation 인프라 레이어다. 시스템은 두 가지 역할을 갖는다: (1) 액션 마스터 테이블 — 키보드/마우스와 게임패드 바인딩을 포함한 모든 액션을 `InputMap`에 등록하고 게임 시작 시 초기화한다; (2) 런타임 리바인딩 — 플레이어가 메인 메뉴/설정 UI를 통해 바인딩을 수정하면 변경 사항을 `user://input_bindings.cfg`에 영속 저장하고 다음 세션에 복원한다. 다운스트림 시스템(플레이어 캐릭터 컨트롤러, UI)은 이 시스템이 보장하는 액션 이름 계약에만 의존하며 물리적 키를 직접 참조하지 않는다. 지원 입력: 키보드/마우스(기본), 게임패드(Steam Deck 호환 — 모든 핵심 액션에 게임패드 바인딩 필수).

## Player Fantasy

입력이 맞을 때, 플레이어는 버튼을 잊는다.

PC로 늦은 밤 유랑단을 이끌 때 — 지난주에 공격 키를 왼손이 편한 위치로 바꿨고, 이제 그냥 된다. 게임이 낡은 재킷처럼 몸에 맞는다. 동료들이 서로를 알아가듯, 게임이 내 손을 배운 것처럼. 그리고 그 순간 이 시스템은 사라진다.

입력 매핑 시스템은 Pillar 1 (모인 사람들)과 Pillar 2 (눈에 보이는 성장)의 보이지 않는 수문장이다. 실시간 파티 전투에서 입력에 마찰이 생기는 순간, 유랑단은 "나 + NPC 몇 명"으로 분해된다. 이 시스템이 제대로 동작할 때 플레이어는 그것의 존재를 모른다 — 그들은 자신의 팀과 함께 있을 뿐이다.

> **MVP 범위 명시**: 이 시스템은 키보드/마우스 리바인딩을 지원한다. 게임패드 리바인딩은 Post-MVP로 연기됨 (OQ-1 — **Fantasy Gap**: 게임패드 커스터마이징 판타지는 Post-MVP에서 완성된다). 동료 명령 액션은 동료 AI 시스템 GDD에서 별도 정의되며, 이 시스템은 플레이어 조작 14개 액션만 관리한다.

## Detailed Design

### Core Rules

**R-1: 단일 소유권**
`InputMapManager` (Autoload Singleton)가 `InputMap`을 소유한다. 다른 어떤 노드도 `InputMap`을 직접 수정하지 않는다.

**R-2: 액션 마스터 테이블**

| 상수 | 액션 이름 | **카테고리** | 기본 키보드 | 기본 게임패드 | 필수 |
|------|-----------|------------|------------|--------------|------|
| `MOVE_UP` | `move_up` | `gameplay` | W / ↑ | 왼쪽 스틱 위 / D-Pad 위 | ✓ |
| `MOVE_DOWN` | `move_down` | `gameplay` | S / ↓ | 왼쪽 스틱 아래 / D-Pad 아래 | ✓ |
| `MOVE_LEFT` | `move_left` | `gameplay` | A / ← | 왼쪽 스틱 왼쪽 / D-Pad 왼쪽 | ✓ |
| `MOVE_RIGHT` | `move_right` | `gameplay` | D / → | 왼쪽 스틱 오른쪽 / D-Pad 오른쪽 | ✓ |
| `COMBAT_ATTACK` | `combat_attack` | `gameplay` | J / 마우스 좌클릭 | 버튼 South (×/A) | ✓ |
| `COMBAT_DODGE` | `combat_dodge` | `gameplay` | K / Space | 버튼 East (○/B) | ✓ |
| `INTERACT` | `interact` | `gameplay` | F | 버튼 North (△/Y) | ✓ |
| `GAME_PAUSE` | `game_pause` | `ui`* | Escape | Start | ✓ |
| `GAME_CONFIRM` | `game_confirm` | `ui` | Enter | 버튼 South (×/A) | ✓ |
| `GAME_CANCEL` | `game_cancel` | `ui` | Escape / Backspace | 버튼 East (○/B) | ✓ |
| `GAME_NAV_UP` | `game_nav_up` | `ui` | ↑ / W | 왼쪽 스틱 위 / D-Pad 위 | ✓ |
| `GAME_NAV_DOWN` | `game_nav_down` | `ui` | ↓ / S | 왼쪽 스틱 아래 / D-Pad 아래 | ✓ |
| `GAME_NAV_LEFT` | `game_nav_left` | `ui` | ← / A | 왼쪽 스틱 왼쪽 / D-Pad 왼쪽 | ✓ |
| `GAME_NAV_RIGHT` | `game_nav_right` | `ui` | → / D | 왼쪽 스틱 오른쪽 / D-Pad 오른쪽 | ✓ |

> **카테고리 규칙**: `gameplay` 액션은 `InputMapManager.is_ui_active = true` 시 PlayerController에서 처리 차단 (R-7). `ui` 액션은 항상 활성.
> *`game_pause`(ui) 예외: `is_ui_active = true` 시 처리 비활성화 — UI 씬에서 Escape = `game_cancel` 전담, `game_pause` 미발동 (R-7).
> **중복 감지 범위**: F-4 중복 검사는 같은 카테고리 내에서만 실행. 카테고리 간 충돌(`combat_attack`/`game_confirm` JOY_BUTTON_SOUTH 공유 등)은 R-7 컨텍스트 분리로 처리.
> **새 액션 추가 시**: 반드시 `category` 컬럼을 명시할 것 — 암묵적 명칭 규칙에 의존하지 않는다.
>
> **설계 메모**: Godot 내장 `ui_accept` / `ui_cancel` 등과 충돌 방지를 위해 게임 전용 UI 액션은 `game_` 접두사를 사용한다. 모든 액션 상수는 `InputActions.gd` Autoload에 정의한다. 게임패드 Left Stick: deadzone 0.2.

**R-3: 초기화 흐름**

```
게임 시작
│
├─ [1단계] 프로젝트 설정 기본값으로 InputMap 로드 (항상 실행)
│
├─ [2단계] user://input_bindings.cfg 존재 확인
│       없음 → 기본값 그대로 사용, 초기화 완료
│       있음 → [3단계]
│
└─ [3단계] 파일 로드 및 액션별 덮어쓰기
        파싱 실패 → 기본값 사용, 경고 메시지 표시
        각 액션: 화이트리스트 검증 → 이벤트 타입 검증 → 통과 시 교체, 실패 시 해당 액션만 기본값 유지
        완료 → signal input_map_ready 발행
```

모든 다운스트림 시스템은 `input_map_ready` 신호 이후에 입력을 처리한다.

**시그널 연결 패턴 (Autoload 타이밍 레이스 방지):**

Autoload는 씬보다 먼저 초기화되므로, 동적으로 로드된 씬은 시그널 발행 이후에 연결될 수 있다. `InputMapManager`는 `var is_ready: bool = false`를 소유하며, 초기화 완료 시 `true`로 설정 후 시그널 발행. 모든 다운스트림 노드는 아래 패턴을 사용한다:

```gdscript
func _ready() -> void:
    if InputMapManager.is_ready:
        _on_input_map_ready()
    else:
        InputMapManager.input_map_ready.connect(_on_input_map_ready)
```

**R-4: 리바인딩 유효성 검사**

리바인딩 요청은 `InputMapManager.request_rebind(action, new_event)` 단일 함수로만 처리한다. 검사 순서:

1. `action`이 마스터 테이블에 존재하는가? → 아니면 `UNKNOWN_ACTION` 거부
2. `new_event`가 허용 타입인가? (InputEventKey, InputEventMouseButton, InputEventJoypadButton, InputEventJoypadMotion) → 아니면 `INVALID_EVENT_TYPE` 거부
3. 예약된 키인가? (F1~F12, Win/Cmd, PrintScreen) → `RESERVED_KEY` 거부
4. 같은 카테고리 내 다른 액션과 중복인가? → `DUPLICATE_BINDING` 반환 (사용자 확인 요청)
5. 모두 통과 → InputMap 업데이트 → 즉시 저장 → `SUCCESS` (저장 실패 시 `SAVE_FAILED` — InputMap은 메모리에 적용됨)

**R-5: 중복 바인딩 처리 (UX)**

충돌 발생 시 팝업 없이 인라인으로 처리한다:
- 충돌 행을 하이라이트, 한 줄 텍스트 표시: *"이 키는 이미 [액션명]에 사용됩니다. 교체하면 [액션명]은 미할당됩니다."*
- 플레이어가 확인 → 기존 액션에서 키 제거 후 새 액션에 할당 (강제 스왑)
- 필수 액션은 최소 바인딩 1개를 유지해야 한다 (마지막 바인딩 삭제 불가)

**R-6: 저장/복원**

저장 경로: `user://input_bindings.cfg` (Godot ConfigFile 형식)

| 타이밍 | 동작 |
|--------|------|
| 리바인딩 확정 즉시 | 저장 |
| 기본값 초기화 시 | 파일 삭제 (파일 없음 = 기본값) |
| 게임 종료 시 | 저장 안 함 (이미 확정 시점에 저장됨) |

InputEvent 객체는 직접 저장 불가 — primitive 필드로 분해 후 저장 (keycode, button_index, axis 등). `physical_keycode` 우선 저장 (비-QWERTY 키보드 이식성).

**R-7: 컨텍스트 분리**

게임패드에서 `combat_attack`과 `game_confirm`이 같은 버튼을 공유하고, Escape 키가 `game_pause`와 `game_cancel` 양쪽에 바인딩되므로 명시적 컨텍스트 관리가 필요하다.

**소유권**: `InputMapManager`가 `var is_ui_active: bool = false` 플래그를 소유한다.

**UI 진입 시** (`set_ui_active(true)` 호출 — 씬 전환 시스템이 담당):
- `gameplay` 카테고리 액션 처리 차단 — PlayerController가 가드로 구현 (아래)
- `game_pause` 액션 처리 비활성화 — UI 씬에서 Escape = `game_cancel` 전담
- Control 노드 이벤트 차단: `focus_mode = FOCUS_NONE`, `mouse_filter = MOUSE_FILTER_IGNORE` (게임플레이 씬)

**UI 이탈 시** (`set_ui_active(false)` 호출):
- `gameplay` 카테고리 액션 처리 재활성화
- `game_pause` 액션 처리 재활성화

**다운스트림 코드 의무**:
- `PlayerController._physics_process()` 진입 시 가드 필수:
  ```gdscript
  func _physics_process(delta: float) -> void:
      if InputMapManager.is_ui_active:
          return
      handle_input()
  ```
- 모든 입력 참조는 `Input.is_action_pressed(InputActions.COMBAT_ATTACK)` 형태만 사용 — 물리 키코드 하드코딩 금지

> **설계 근거**: `PROCESS_MODE_DISABLED`는 `_process`/`_physics_process` 콜백만 차단하며 `Input` 싱글턴 폴링(`Input.is_action_pressed()`)을 차단하지 않는다. 폴링 가드는 PlayerController 레이어에서 명시적으로 구현해야 한다.

### States and Transitions

| 상태 | 설명 | 진입 조건 | 탈출 조건 |
|------|------|-----------|-----------|
| `INITIALIZING` | 기본값 등록 + 저장 파일 로드 | 게임 시작 | 초기화 완료 |
| `ACTIVE` | 정상 입력 처리 중 | 초기화 완료 | 리바인딩 시작 또는 게임 종료 |
| `AWAITING_INPUT` | 플레이어의 새 키 입력 대기 | 설정 UI에서 항목 선택 | 키 입력 받음 또는 ESC |
| `SAVING` | ConfigFile 쓰기 중 | 리바인딩 확정 | 쓰기 완료 또는 실패 |

```
INITIALIZING → ACTIVE → AWAITING_INPUT → SAVING → ACTIVE
                 ↑                          ↓ (ESC)
                 └──────────────────────────┘
```

`AWAITING_INPUT` 중: ESC = 취소 후 `ACTIVE` 복귀. 현재 활성 입력 모드(키보드/게임패드)의 입력만 수신.

### Interactions with Other Systems

| 시스템 | 방향 | 인터페이스 |
|--------|------|-----------|
| **플레이어 캐릭터 컨트롤러** | 다운스트림 (이 시스템이 제공) | `InputActions.*` 상수로 액션 이름 참조. `input_map_ready` 신호 이후 입력 처리 시작. |
| **메인 메뉴 / 설정 UI** | 다운스트림 (이 시스템이 제공) | `InputMapManager.request_rebind(action, event)` 호출, `RebindResult` enum 수신. 현재 바인딩 조회: `InputMap.action_get_events(action)`. 기본값 초기화: `InputMapManager.reset_to_defaults()`. |
| **씬 전환 시스템** | 상호작용 없음 | 입력 매핑은 씬 전환과 독립적. 단, 씬 전환 중 컨텍스트 분리(R-7)는 씬 전환 시스템이 `process_mode`를 제어해 구현. |

## Formulas

### F-1: 게임패드 축 → 액션 발동 판정 (Deadzone Threshold)

```
is_active = |axis_value| > deadzone
```

| 변수 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `axis_value` | float | [-1.0, 1.0] | Godot InputEventJoypadMotion.axis_value |
| `deadzone` | float | [0.0, 1.0] | 상수 = 0.2 (DEADZONE_LEFT_STICK) |

**출력 범위**: bool — true이면 해당 move_* 액션 발동
**예시**: axis_value=0.15 → false (드리프트 무시) / axis_value=0.25 → true (발동)

> `Input.get_vector()` 사용 시 deadzone 파라미터로 직접 전달 가능.

---

### F-2: 이동 벡터 계산 (Radial Deadzone + 정규화)

**구현**: Godot 관용 패턴 `Input.get_vector()` 사용 — 원형(radial) 데드존 및 정규화 자동 처리.

```
move_vector = Input.get_vector(&"move_left", &"move_right", &"move_up", &"move_down", DEADZONE_LEFT_STICK)
```

| 변수 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `DEADZONE_LEFT_STICK` | float | [0.0, 1.0] | 데드존 = 0.2 |

**알고리즘**: 원형(radial) 데드존 — 입력 벡터 크기 > `deadzone`이면 정규화된 Vector2 반환; 이하이면 Vector2.ZERO.

**출력 범위**: Vector2, 길이 0.0~1.0 / 무입력 시 Vector2.ZERO
**예시**: (0.7, 0.7) → normalized (0.707, 0.707) — 대각선 속도 동일 보장

> **원형 vs 성분별 데드존**: `get_vector()`는 원형 데드존 적용. 입력 (0.15, 0.15) 시: 크기 ≈ 0.212 > 0.2 → 대각선 (0.707, 0.707) 반환 (자연스러운 전환). 성분별 방식과 달리 데드존 경계에서 직선 스냅이 없다.
> StringName 리터럴 (`&"action_name"`) 사용으로 핫패스 String 할당 최적화.
> F-1의 `is_active` 공식은 단일 축 액션 트리거 판정 전용; F-2는 이동 벡터 계산 전용 — 별개의 코드 경로.

### F-3: 아날로그 스틱 → 4방향 이산화

```
magnitude = Vector2(axis_x, axis_y).length()
angle_deg = atan2(axis_y, axis_x) × (180/π)    ← [-180, 180]

magnitude > dz 일 때:
  -45° < angle ≤  45°                            → move_right
   45° < angle ≤ 135°                            → move_down
  135° < angle ≤ 180° 또는 -180° ≤ angle ≤ -135° → move_left
 -135° < angle ≤ -45°                            → move_up
magnitude ≤ dz → NONE
```

| 변수 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `axis_x` | float | [-1.0, 1.0] | Left Stick X (오른쪽 양수) |
| `axis_y` | float | [-1.0, 1.0] | Left Stick Y (아래쪽 양수) |
| `dz` | float | [0.0, 1.0] | deadzone = 0.2 |
| `magnitude` | float | [0.0, √2] | 입력 벡터 크기 |

**출력 범위**: enum {NONE, UP, DOWN, LEFT, RIGHT} — 단 하나의 방향 (동시 발동 없음)
**예시**: (0.9, 0.1) → angle=6.3° → move_right / (0.0, -0.9) → angle=-90° → move_up

> **구현 주의**: magnitude 검사를 먼저 실행. `atan2(0, 0) = 0° → move_right`가 되므로 `magnitude ≤ dz` 조건에서 각도 계산 없이 즉시 NONE을 반환할 것. `Vector2(axis_x, axis_y).angle()` 관용 패턴 사용 권장 (라디안 반환 후 변환).
> 45° 경계 정책: `DIAGONAL_THRESHOLD_DEG = 45.0` 상수로 관리. 정확히 45°는 move_right 포함.
> D-Pad 물리 입력은 이미 이산 신호이므로 이 공식은 Left Stick에만 적용.
> **좌표계 주의**: Godot Y-down 기준. 양수 axis_y = 아래쪽. angle 90° = move_down, -90° = move_up.

---

### F-4: 리바인딩 중복 감지 판정

```
InputEventKey:          physical_keycode 일치 AND ctrl_pressed 일치 AND shift_pressed 일치 AND alt_pressed 일치
InputEventMouseButton:  button_index 일치
InputEventJoypadButton: button_index 일치
InputEventJoypadMotion: axis 일치 AND sign(axis_value) 일치
타입 다름:              false (키보드 J ≠ 게임패드 South — 크로스 타입은 중복 아님)
```

| 변수 | 타입 | 범위 | 설명 |
|------|------|------|------|
| `candidate_event` | InputEvent | — | 리바인딩 요청으로 들어온 새 이벤트 |
| `existing_event` | InputEvent | — | 다른 액션에 등록된 기존 이벤트 |
| `physical_keycode` | int | Key enum | 레이아웃 독립 물리 키코드 |
| `axis_value sign` | int | {-1, +1} | 방향만 구분 (크기 무관) |

**출력 범위**: bool — true이면 DUPLICATE_BINDING, 사용자 확인 요청
**검사 범위**: 같은 카테고리(gameplay/ui) 내 전체 액션 — 카테고리 정의는 R-2 마스터 테이블의 `category` 컬럼 참조. 카테고리 간 중복(gameplay ↔ ui)은 R-7 컨텍스트 분리로 처리.

> **E-2 의존성**: 이 공식은 `InputEventJoypadMotion.axis_value = 0.0` 이벤트가 도달하지 않는다고 가정한다 (`sign(0.0) = 0`이 세 번째 부호값으로 작용해 중복 미감지 유발). E-2의 `REBIND_AXIS_THRESHOLD(0.5)` 가드가 AWAITING_INPUT 단계에서 해당 이벤트를 미리 차단하므로 F-4는 그 이후에 실행된다.

## Edge Cases

### E-1: 초기화 시점

- **If** 저장 파일의 schema_version이 현재보다 낮거나 없는 경우: 파일 전체를 무시하고 기본값으로 초기화. 파일은 보존하며 경고 로그 출력. `input_map_ready` 정상 발행.

- **If** 저장 파일에 현재 마스터 테이블에 없는 액션이 있는 경우: 해당 섹션만 건너뜀. 알려진 액션들은 정상 복원. 경고 로그 출력 (조용한 복구).

- **If** InputMapManager 외 다른 Autoload가 InputMap을 직접 수정하려 하는 경우: R-1 위반. 자기 자신의 중복 초기화는 불리언 가드로 차단. 외부 수정은 Forbidden Pattern으로 등록 후 초기화 완료 시 액션 목록 검증으로 사후 감지.

### E-2: 리바인딩 중

- **If** AWAITING_INPUT 상태에서 OS 창 포커스를 잃는 경우(Alt+Tab): 리바인딩 취소, ACTIVE 복귀. InputMap 변경 없음. [이유: 포커스 복귀 시 Alt/Tab key-up 이벤트 누락으로 의도치 않은 바인딩 등록 방지.]

- **If** 현재 바인딩과 동일한 키를 다시 누른 경우: 변경 없이 SUCCESS 반환, ACTIVE 복귀. 파일 저장 없음.

- **If** 스틱 틸트가 REBIND_AXIS_THRESHOLD(0.5) 미만인 경우: 이벤트 무시, AWAITING_INPUT 유지. [이유: 게임플레이 deadzone(0.2)과 구분 — 드리프트로 인한 의도치 않은 바인딩 방지.]

- **If** 필수 액션의 마지막 바인딩을 삭제하려 하는 경우: LAST_BINDING_BLOCKED 거부. UI는 바인딩이 1개일 때 삭제 버튼 비활성화.

- **If** AWAITING_INPUT 중 Escape 키를 누른 경우: 리바인딩 취소 후 ACTIVE 복귀. **반드시 raw `InputEventKey.physical_keycode == KEY_ESCAPE` 비교 사용** — `Input.is_action_just_pressed("game_cancel")` 액션 시스템 사용 금지. [이유: 액션 기반 취소는 (1) settings screen을 함께 닫을 수 있고, (2) Escape 키 자체를 리바인딩하려는 경우 의도치 않은 취소를 일으킨다.]

- **If** AWAITING_INPUT 중 입력 기기가 전환된 경우 (예: 키보드 모드 중 게임패드 입력): 이벤트 무시, AWAITING_INPUT 유지. `InputMapManager`는 `current_input_mode: InputMode` enum(KEYBOARD, GAMEPAD)으로 마지막 활성 기기를 추적하며, AWAITING_INPUT 진입 시 모드를 고정한다. [MVP OQ-1: 게임패드 리바인딩 미지원이므로 현재는 키보드/마우스 모드만 AWAITING_INPUT 진입 가능.]

### E-3: 저장/복원

- **If** 저장 중 강제 종료로 파일이 부분 기록된 경우: 다음 시작 시 파싱 실패 감지 → 기본값 사용 + 손상 파일 삭제 + 경고 표시. [Post-MVP: 임시 파일 rename 패턴으로 개선 권장.]

- **If** 파일 쓰기 권한이 없는 경우: InputMap은 메모리에 적용됨 (현재 세션 유효). UI에 비차단 경고 표시: "저장 불가 — 이 세션에만 적용됩니다." 게임 플레이는 중단하지 않음.

### E-4: 런타임 입력

- **If** 게임패드가 플레이 중 연결 해제되는 경우: Godot가 해당 기기 입력 상태를 자동 클리어. 캐릭터는 다음 프레임에 즉시 정지. InputMapManager 추가 처리 없음. 키보드는 즉시 사용 가능.

- **If** 두 개의 게임패드가 동시 연결된 경우: 1인 싱글 플레이어 게임이므로 두 패드 모두 입력 수신 (Godot 기본 동작). 동시 반대 방향 입력 시 축 합산 후 클램핑. 기기 잠금(device lock) 기능은 MVP 범위 외.

## Acceptance Criteria

### AC-R1: 단일 소유권

**GIVEN** `InputMapManager`가 ACTIVE 상태에서, **WHEN** 테스트 스텁이 `InputMap.action_erase_events("move_up")`을 호출한 뒤 `InputMap.action_add_event("move_up", InputEventKey.new() with keycode=KEY_Z)`를 직접 호출(외부 무단 수정 시뮬레이션), **THEN** 다음 중 하나가 성립한다: (a) R-1이 런타임 검증으로 구현된 경우: `InputMapManager._validate_action_integrity()` 실행 후 `action_get_events("move_up")`에 KEY_Z가 없고 기본값 복원; (b) R-1이 정적 분석으로만 구현된 경우: AC-R7의 grep 검사가 이 제약을 커버하며, 해당 테스트 스텁 코드는 `src/` 외부에 위치.

> **구현 노트**: R-1이 Forbidden Pattern(정적 분석) 방식이면 이 AC를 AC-R7의 grep 범위에 통합할 것.

### AC-R2: 액션 마스터 테이블

**GIVEN** 신규 설치 첫 실행 (`input_bindings.cfg` 없음), **WHEN** `input_map_ready` 발행 직후, **THEN** 14개 액션 이름 전부에 대해 `InputMap.has_action()`이 `true`를 반환한다.

**GIVEN** 게임 초기화 후, **WHEN** `InputMap.action_get_events(action)`을 14개 액션 각각에 호출할 때, **THEN** 모든 액션에 `InputEventJoypadButton` 또는 `InputEventJoypadMotion` 이벤트가 최소 1개 이상 존재한다.

**GIVEN** 게임 초기화 후, **WHEN** `InputMap.action_get_events("game_confirm")`과 `InputMap.action_get_events("ui_accept")`를 비교할 때, **THEN** 두 목록에 동일한 이벤트가 단 하나도 없다 (게임 전용 액션과 Godot 내장 액션 간 이벤트 중복 = 0).

### AC-R3: 초기화 흐름

**GIVEN** `input_bindings.cfg`가 없을 때 시작, **WHEN** 초기화 완료, **THEN** (1) `move_up`이 W/↑ 키 이벤트 포함, (2) `input_map_ready`가 정확히 1회 발행, (3) `InputMapManager`의 `_ready()` / `_initialize()` 실행 경로에서 `push_warning()` 호출 횟수 = 0 (스파이/mock으로 검증).

**GIVEN** `move_up`이 Z키로 저장된 cfg 존재, **WHEN** `input_map_ready` 직후 조회, **THEN** `InputMap.action_get_events("move_up")`에 `InputEventKey(physical_keycode=KEY_Z)` 포함.

**GIVEN** 손상된 cfg 파일 존재, **WHEN** 초기화 완료, **THEN** (1) 기본값 사용, (2) `input_map_ready` 1회 발행, (3) 경고 로그에 파싱 실패 메시지 1회 이상.

**GIVEN** 플레이어 컨트롤러 씬이 로드되고 `InputMapManager`가 INITIALIZING 상태(`is_ready = false`)인 동안, **WHEN** `Input.action_press("move_up")`으로 키 입력 강제 주입 후 `_physics_process`가 1회 실행, **THEN** `PlayerController._get_input_vector()` (또는 동등한 입력 처리 함수)가 `Vector2.ZERO`를 반환하며, 이 결과가 "INITIALIZING 가드가 작동해서"임을 확인하기 위해 `InputMapManager.is_ready = true` 설정 후 동일 조건에서는 Vector2.ZERO가 아닌 값이 반환되어야 한다 (가드 유무 대비 테스트).

### AC-R4: 리바인딩 유효성 검사

**GIVEN** ACTIVE 상태, **WHEN** `request_rebind("fly_up", event)` (마스터 테이블 없는 이름), **THEN** `RebindResult.UNKNOWN_ACTION`, InputMap 불변.

**GIVEN** ACTIVE 상태, **WHEN** `request_rebind("move_up", InputEventMIDI.new())`, **THEN** `RebindResult.INVALID_EVENT_TYPE`, InputMap 불변.

**GIVEN** ACTIVE 상태, **WHEN** F1~F12, Win/Cmd, PrintScreen 중 하나로 `request_rebind`, **THEN** `RebindResult.RESERVED_KEY`, InputMap 불변.

**[로직]** **GIVEN** `combat_attack`이 J 키 바인딩, **WHEN** `request_rebind("interact", InputEventKey(physical_keycode=KEY_J))`, **THEN** 반환값 = `RebindResult.DUPLICATE_BINDING` AND `InputMap.action_get_events("interact")`에 KEY_J 없음. InputMap 미변경.

**[UI/통합]** **GIVEN** 리바인딩 설정 화면이 열린 상태, `combat_attack`이 J, **WHEN** 플레이어가 `interact` 행에서 J 키 입력, **THEN** 충돌 행 하이라이트 AND 인라인 텍스트 표시 (내용: "이 키는 이미 [combat_attack 표시명]에 사용됩니다...").

**GIVEN** ACTIVE 상태, cfg 없음, **WHEN** `request_rebind("interact", KEY_E)`, **THEN** (1) `RebindResult.SUCCESS`, (2) `action_get_events("interact")`에 KEY_E 포함, (3) cfg에 KEY_E physical_keycode 저장 — 동일 프레임 내 완료.

### AC-R5: 중복 처리

**GIVEN** `combat_attack`이 J 단독 바인딩, 사용자가 `interact`에 J 강제 스왑 확인, **WHEN** 두 액션 이벤트 조회, **THEN** `combat_attack`에서 KEY_J 제거, `interact`에 KEY_J 추가.

**[로직]** **GIVEN** `move_up` 바인딩이 W 1개만 존재, **WHEN** 해당 바인딩 삭제 시도, **THEN** 반환값 = `RebindResult.LAST_BINDING_BLOCKED` AND `InputMap.action_get_events("move_up")`에 여전히 W 키 존재.

**[UI]** **GIVEN** 리바인딩 화면에서 `move_up`이 바인딩 1개만 표시될 때, **THEN** 해당 행의 삭제 버튼 `disabled = true` (씬 트리 검사 또는 수동 워크스루 문서로 검증).

### AC-R6: 저장/복원

**GIVEN** ACTIVE 상태, **WHEN** `request_rebind("combat_dodge", KEY_L)`이 SUCCESS 반환 직후, **THEN** cfg를 `ConfigFile.load()`로 읽으면 `combat_dodge` 섹션에 KEY_L physical_keycode 값 존재.

**GIVEN** 커스텀 cfg 존재, **WHEN** `reset_to_defaults()` 호출, **THEN** `FileAccess.file_exists("user://input_bindings.cfg")` = `false`.

**GIVEN** `request_rebind("interact", event)` 호출 시 `event.physical_keycode = KEY_E (69)` AND `event.keycode = KEY_Q (81)` (AZERTY 논리 키 시뮬레이션), **WHEN** SUCCESS 반환 후 cfg 파일 확인, **THEN** `ConfigFile.get_value("interact", "physical_keycode")` = 69 AND `keycode` 필드는 저장되지 않음 (또는 존재 시 0). [하드웨어 환경 불필요 — 이벤트 객체를 직접 생성해 테스트]

### AC-R7: 컨텍스트 분리

**GIVEN** 게임플레이 중, **WHEN** 설정 UI 진입으로 `InputMapManager.set_ui_active(true)` 호출, **THEN** 게임패드 South 버튼 입력 시: (1) PlayerController의 `handle_input()` 미실행 (is_ui_active 가드 작동), (2) `Input.is_action_pressed("game_confirm")` = true (ui 액션은 여전히 활성).

**GIVEN** `InputMapManager.set_ui_active(true)` 상태, **WHEN** Escape 키 입력, **THEN** `Input.is_action_just_pressed("game_cancel")` = true AND `Input.is_action_just_pressed("game_pause")` = false (`game_pause` 비활성화 확인).

**[음성 케이스 — 회귀 앵커]** **GIVEN** `set_ui_active()` 호출 없이 설정 UI를 진입한 시뮬레이션(is_ui_active = false 유지), **WHEN** 게임패드 South 버튼 입력, **THEN** PlayerController의 `handle_input()`이 실행됨 — 컨텍스트 분리가 `is_ui_active` 플래그 메커니즘에 의존함을 확인.

**GIVEN** `src/` 디렉터리 하위 전체 `.gd` 파일 (`src/autoloads/input_actions.gd` 및 `tests/` 경로 제외), **WHEN** 정적 분석 — `KEY_[A-Z0-9_]+`, `JOY_BUTTON_[A-Z0-9_]+`, `JOY_AXIS_[A-Z0-9_]+` 패턴이 `Input.is_action`, `InputMap.` 호출 외부에서 사용된 경우, **THEN** 0건. [참고: Godot 4.x에서 `JOY_BUTTON_0`은 존재하지 않음 — `JOY_BUTTON_SOUTH` 등 이름 상수 사용]

### AC-F1: Deadzone

**GIVEN** `DEADZONE_LEFT_STICK = 0.2`, **WHEN** Left Stick X에 `axis_value = 0.19` 주입, **THEN** `Input.is_action_pressed("move_right")` = false.

**GIVEN** `DEADZONE_LEFT_STICK = 0.2`, **WHEN** Left Stick X에 `axis_value = 0.20` 주입, **THEN** `Input.is_action_pressed("move_right")` = false. [경계값 엄격한 `>` 검증 — `>=` 구현 감지]

**GIVEN** `DEADZONE_LEFT_STICK = 0.2`, **WHEN** Left Stick X에 `axis_value = 0.21` 주입, **THEN** `Input.is_action_pressed("move_right")` = true.

### AC-F2: 이동 벡터 정규화

**GIVEN** Left Stick `(0.7, 0.7)` 입력, **WHEN** 이동 벡터 계산, **THEN** `move_vector.length()` = 1.0 (±0.001).

**GIVEN** 모든 축 0.0, **WHEN** 이동 벡터 계산, **THEN** `move_vector == Vector2.ZERO`.

### AC-F3: 4방향 이산화

**GIVEN** `DIAGONAL_THRESHOLD_DEG = 45.0`, **WHEN** Left Stick angle = 44.9° (magnitude > 0.2), **THEN** `move_right` = true, `move_down` = false.

**GIVEN** `DIAGONAL_THRESHOLD_DEG = 45.0`, **WHEN** Left Stick angle = 45.0° (경계값, magnitude > 0.2), **THEN** `move_right` = true, `move_down` = false. [정확히 45°는 move_right 포함]

**GIVEN** angle = 45.1° (magnitude > 0.2), **THEN** `move_down` = true, `move_right` = false.

**GIVEN** angle = 134.9° (magnitude > 0.2), **THEN** `move_down` = true, `move_left` = false.

**GIVEN** angle = 135.0° (magnitude > 0.2), **THEN** `move_left` = true. [135° 경계 포함]

**GIVEN** angle = -135.0° (magnitude > 0.2), **THEN** `move_left` = true. [음수 경계 포함]

**GIVEN** angle = -134.9° (magnitude > 0.2), **THEN** `move_up` = true, `move_left` = false.

**GIVEN** angle = 0°/90°/180°/-90° 각각 입력, **WHEN** 방향 액션 확인, **THEN** move_right/move_down/move_left/move_up 중 해당 방향 1개만 true.

### AC-F4: 중복 감지 판정

**GIVEN** `combat_attack`이 Ctrl+J 바인딩, **WHEN** `request_rebind("interact", KEY_J, ctrl=true)`, **THEN** `RebindResult.DUPLICATE_BINDING`.

**GIVEN** `combat_attack`이 J 바인딩 (modifier 없음), **WHEN** `request_rebind("interact", KEY_J, ctrl=true)`, **THEN** `RebindResult.SUCCESS` (J ≠ Ctrl+J).

**GIVEN** `combat_attack`이 K 키 바인딩, **WHEN** `request_rebind("interact", InputEventJoypadButton(JOY_BUTTON_SOUTH))`, **THEN** `RebindResult.SUCCESS` (타입 달라 중복 아님).

### AC-E: 주요 엣지 케이스

**GIVEN** cfg에 `schema_version = 0` (구버전, 현재 코드의 `CURRENT_SCHEMA_VERSION`보다 낮은 값), **WHEN** 게임 시작 후 `input_map_ready`, **THEN** (1) 14개 액션 기본값, (2) 파일 보존, (3) `push_warning()` 호출에 schema 불일치 메시지 포함.

**GIVEN** 신규 설치(cfg 없음)에서 `request_rebind("interact", KEY_E)`가 SUCCESS 반환 후, **WHEN** `ConfigFile.load("user://input_bindings.cfg")`, **THEN** `[meta]` 섹션(또는 동등 섹션)의 `schema_version` 값 ≥ 1 (현재 코드의 `CURRENT_SCHEMA_VERSION` 값과 일치). [이 테스트가 없으면 schema_version=0인 코드는 E-1의 구버전 감지 테스트를 항상 통과시킴]

**GIVEN** `AWAITING_INPUT` 상태 (`interact` 리바인딩 대기), **WHEN** OS 포커스 손실, **THEN** 상태 = ACTIVE, `action_get_events("interact")` 불변, cfg 불변.

**GIVEN** `user://` 쓰기 권한 없음, **WHEN** `request_rebind("move_up", KEY_T)`, **THEN** (1) `Input.is_action_pressed("move_up")`이 T키 반응 (메모리 적용), (2) UI 비차단 경고 표시, (3) 반환값 = `RebindResult.SAVE_FAILED`.
